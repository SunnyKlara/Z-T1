## 一、核心原则

### RideWind 的教训

| 问题 | 表现 | 后果 |
|------|------|------|
| 没有阶段划分 | 想到什么写什么 | 代码臃肿、职责混乱 |
| 没有验证节点 | 写完一堆再测 | 问题堆积、难以定位 |
| 没有文档先行 | 先写代码后补文档 | 两端不一致、协议混乱 |
| 没有行数约束 | 文件无限膨胀 | 可读性差、难以维护 |
| 没有交接机制 | 对话之间信息丢失 | 重复讨论、方向偏离 |

### ZCritical 的规则

| 规则 | 含义 |
|------|------|
| **文档先行** | 任何代码修改前，先确认文档已更新 |
| **阶段隔离** | 每个阶段只做一件事，做完验证再进入下一个 |
| **单点验证** | 每写完一个文件 → 编译/分析通过 → 再写下一个 |
| **行数硬约束** | 每个文件有上限，超标必须拆分 |
| **对话交接** | 每个对话结束更新 session-handoff，下一个对话从 handoff 开始 |
| **契约驱动** | 协议变更先改 protocol-contract.md，再改两端代码 |

## 二、开发节奏

### 标准开发循环

```
1. 读文档 (5分钟)
   └─ 读 global-development-roadmap.md → 确认当前阶段
   └─ 读对应端 session-handoff.md → 确认上次做到哪
   └─ 读 protocol-contract.md → 确认协议格式

2. 出设计契约 (10分钟)
   └─ 列出要创建/修改的文件
   └─ 每个文件的职责、行数上限
   └─ 用户确认

3. 执行 (按文件逐个)
   └─ 写文件 → 编译/分析 → 通过 → 下一个
   └─ 不跳步、不并行、不提前写后面的

4. 验证 (5分钟)
   └─ flutter analyze / idf.py build
   └─ 检查行数
   └─ 检查 STEER 块

5. 交接 (5分钟)
   └─ 更新 session-handoff.md
   └─ 输出合规检查报告
```

### 禁止的行为

| 禁止 | 为什么 |
|------|--------|
| 一次写多个文件再一起编译 | 问题堆积，不知道哪个文件导致的 |
| 跳过设计契约直接写代码 | 方向可能偏，写了也白写 |
| 文件超过行数上限不拆分 | 可读性差，违反反臃肿纲领 |
| 不更新文档就改代码 | 文档失效 = 协作崩溃 |
| 对话结束不更新 handoff | 下一个对话不知道从哪开始 |
| 在基建阶段写业务逻辑 | 阶段隔离被打破 |

## 四、智能化规则

### AI 应该自动做的

| 场景 | AI 自动行为 |
|------|------------|
| 打开新对话 | 读 roadmap → 读 handoff → 确认当前阶段 |
| 开始写代码前 | 输出设计契约 → 等待用户确认 |
| 写完一个文件 | 自动运行编译/分析 → 检查行数 → 检查 STEER 块 |
| 文件接近行数上限 | 主动提醒用户需要拆分 |
| 对话结束前 | 更新 handoff → 输出合规报告 |
| 发现协议不一致 | 以 protocol-contract.md 为准，提醒用户 |

### AI 不应该自动做的

| 场景 | AI 应该 |
|------|--------|
| 架构决策变更 | 先和用户讨论，再更新文档 |
| 阶段跳跃 | 提醒用户当前阶段，不建议跳步 |
| 文件拆分方案 | 提供方案让用户选择 |
| 协议变更 | 先更新 protocol-contract.md，再改代码 |
