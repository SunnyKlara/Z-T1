## 一、审计背景

RideWind（旧项目）实现了基本功能闭环，但作为面向市场的嵌入式产品，存在多个级别的缺失。本文档：
1. 列出 RideWind 已实现的功能及其成熟度
2. 按嵌入式产品的市场标准，列出缺失功能
3. 对每个缺失项给出技术方案选型
4. 按优先级分三批，指导开发顺序

## 三、缺失功能清单与技术方案

---

### P0: OTA 固件升级 — 🔴 必须做

**为什么必须做**：没有 OTA 的 IoT 产品无法修复出厂 bug，产品退货率不可控。

**技术方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| BLE OTA | ESP-IDF `esp_ota_begin/write/end`，分片传输 .bin | 后备方案 |
| WiFi OTA | HTTP GET 下载 .bin → OTA 写入 | 主力方案（快100倍） |

**实现要点**：
```
Flash 分区 (8MB):
  factory  → 0x10000~0x310000 (3MB, 出厂固件)
  ota_0    → 0x310000~0x610000 (3MB, OTA槽位0)
  ota_1    → 0x610000~0x910000 (3MB, OTA槽位1)
  剩余     → LittleFS 数据分区

OTA 流程:
  1. APP 检查固件版本 (GET:VERSION)
  2. 服务器对比 → 有更新则下载 .bin
  3. WiFi OTA: 固件端直接下载
     BLE OTA: APP 分片传输 (每包 244 字节有效载荷)
  4. esp_ota_end → 设置启动分区 → 重启
  5. 新固件启动失败 → bootloader 自动回滚到旧分区

安全措施:
  - 下载前校验 CRC32
  - 启动后 30 秒无 crash → 标记固件有效
  - 支持 APP 端查询 OTA 进度
```

**代码量**：固件端 ~200行 + APP端 ~300行

### P1: 看门狗 + 异常恢复 — 🔴 必须做

**为什么必须做**：FreeRTOS 任务死锁/栈溢出 = 设备变砖，嵌入式产品必须有自动恢复能力。

**技术方案 — 三层看门狗架构**：

```
第一层: 硬件看门狗 (TWDT)
  esp_task_wdt_init(10秒) → 超时自动重启
  主任务循环中重置

第二层: 模块级软件看门狗
  BLE 看门狗:   6秒无数据 → 标为断开
  LCD 看门狗:   3秒无刷新 → 重启 LCD 任务
  LED 看门狗:   5秒无更新 → 恢复默认颜色
  音频看门狗:   2秒无数据 → 静音

第三层: Panic Handler
  捕获 crash → 记录: 时间戳 + 任务名 + PC + 回栈 → NVS
  自动重启 → 启动时 LCD 显示"上次异常: [原因]"
```

**代码量**：~80行

---

### P2: 错误日志 + 诊断模式 — 🔴 必须做

**为什么必须做**：用户反馈问题但无法复现时，日志是唯一线索。

**技术方案**：

```
固件端:
  - LittleFS 环形日志缓冲区 (1024 条, 每条 128 字节 = 128KB)
  - 格式: [时间戳] [级别] [模块] 消息
  - 级别: ERROR / WARN / INFO / DEBUG
  - BLE 命令: DUMP_LOG → 导出日志给 APP
  - BLE 命令: GET_STATUS → 返回: 运行时间/堆余量/PSRAM余量/重连次数/BLE状态

APP 端:
  - 诊断页面 (/debug)
  - 仅在版本号点 5 次后可见(开发者模式)
  - 显示: 设备运行时间/崩溃记录/BLE状态/内存余量/固件版本
  - 导出日志到手机文件
```

**代码量**：固件端 ~150行 + APP端 ~100行

---

### P3: 出厂自检 (POST) — 🔴 必须做

**为什么必须做**：每台设备出厂前必须确认所有硬件正常工作，否则退货率无法控制。

**技术方案**：

```
POST 测试序列 (约 30 秒):

  Step 1: NVS / Flash 读写测试
  Step 2: PSRAM 读写测试
  Step 3: LCD 全屏刷色 (白→红→绿→蓝→黑)
  Step 4: LED 灯珠逐一闪烁 (每个灯珠亮 500ms)
  Step 5: 编码器旋转检测 (提示操作员旋转一圈)
  Step 6: 风扇 PWM 最低占空比脉冲 (听声音确认)
  Step 7: 加湿器短脉冲 (视觉确认雾化)
  Step 8: 音频播放测试音 (440Hz 正弦波, 1秒)
  Step 9: BLE 广播验证 (LCD 显示 "请用手机扫描 T1")
  Step 10: 通信测试 (APP 连接后发送 PING → 固件回复 PONG)

  任一失败 → LCD 显示错误码 (如 E03: LCD FAIL)
           → 红灯闪烁 → 不进主循环
  全部通过 → LCD 显示 "PASS" → 写入 NVS 标记 → 进主循环

进入方式:
  - 上电时按住编码器按键 3 秒 → 进入 POST 模式
  - 或 BLE 命令 POST_START
```

**代码量**：~200行

### P4: BLE 连接体验优化 — 🟡 需要改进

**RideWind 问题**：
- 扫描超时 4 秒太短
- 无连接失败原因分类
- 无 BLE 信号强度实时显示

**改进方案**：

| 改进项 | 当前 | 改为 |
|--------|------|------|
| 扫描超时 | 4秒 | 10秒 + 手动停止 |
| 错误分类 | 无 | 蓝牙未开 / 权限未授权 / 未找到 / 超时 / 被拒 — 每条有操作指引 |
| 信号显示 | 无 | BLE Banner 实时显示 RSSI 值 |
| 重连策略 | 指数退避 5次 | 指数退避 5次 + 第6次起改为 60秒间隔 + 手动触发 |

### P5: 音频方案重新选型 — 🔴 核心设计决策

**RideWind 现状**：波表合成 + 程序化合成，ESP32 实时生成引擎声音波形。

**问题分析**：
1. 实时音频合成占用 CPU 40-60%，与其他任务争抢资源
2. 波形合成算法调试极为困难（是声学工程问题，不是代码问题）
3. 合成音质上限低，无法达到"真实引擎声"的效果
4. 调试周期长，且结果不可预测

**三种方案**：

| 方案 | 音质 | CPU占用 | 存储 | 周期 | 扩展性 |
|------|------|---------|------|------|--------|
| A: 实时合成 | ⭐⭐ | 40-60% | ~0 | 2-4周 | 差 |
| B: MP3 样本 + I2S 播放 | ⭐⭐⭐⭐ | 5-10% | ~2MB | 3-5天 | ⭐⭐⭐⭐⭐ |
| C: 混合 (合成基底 + 样本叠加) | ⭐⭐⭐ | 15-25% | ~500KB | 2-3周 | 中 |

**建议：方案 B — MP3 样本播放**。

理由：
- ESP32-S3 + PSRAM + ESP-ADF 的 `esp_audio` 组件原生支持 MP3 解码
- 音质由录音师决定，不依赖算法调试
- CPU 开销极低，不影响其他任务
- 后续可出"音效扩展包" (F1/V8/电动方程式/涡轮/转子引擎...)
- 开发快，风险低

**实现**：
```c
// 使用 esp_audio 组件
#include "esp_audio.h"

// 引擎声音样本 (预录制 MP3, 放入 LittleFS)
// /spiffs/audio/engine_idle.mp3    怠速声
// /spiffs/audio/engine_low.mp3     低速
// /spiffs/audio/engine_mid.mp3     中速
// /spiffs/audio/engine_high.mp3    高速
// /spiffs/audio/engine_start.mp3   启动声
// /spiffs/audio/engine_stop.mp3    熄火声

// 按速度区间切换样本 + 淡入淡出过渡
// 无循环样本 → 静音直到下一个速度区间触发
```

**需要外部资源**：录制或购买 6-8 个高质量引擎声音 MP3 样本（一次性投入）。

### P6: LCD UI 迁移到 LVGL — 🟡 需要改进

**RideWind 现状**：手写点阵字体 UI 渲染 (`ui_common.c/h`)。

**问题**：
- 不支持中文字体
- 不能缩放
- 添加新页面代码量大
- 动画效果有限

**LVGL vs 手写**：

| 维度 | 手写 UI | LVGL |
|------|---------|------|
| 开发效率 | 每个页面 200-400 行 | 50-100 行 |
| 字体 | 自制点阵 | TrueType 任意字号/语言 |
| 动画 | 手写帧动画 | 内置过渡/缩放/渐变 |
| 社区支持 | 无 | 活跃，ESP-IDF 官方组件 |
| ROM占用 | ~20KB | ~200KB |

**决策**：迁移到 LVGL。ESP32-S3 有足够 Flash，200KB 开销完全可接受。

**迁移策略**：逐个页面迁移（先 Speed 页 → Color 页 → RGB 页 → 其他），不一次性替换。

### P7: 数据统计 + 历史记录 — 市场差异化

**产品价值**：让产品从"玩具"升级为"装备"。

**方案**：
```
记录以下数据到 NVS:
  - 总运行时间 / 总风扇时间
  - 各模式的累计使用时长
  - 最常用的 LED 预设 (Top 3)
  - 最高速度记录
  - 固件升级历史 (版本 + 时间)

APP 端:
  - 统计页面展示图表
  - "本周运行了 3.5 小时"
  - "你最喜欢的颜色是 Flame Red"
```

### P8: 场景模式 — 市场差异化

**方案**：
```
预置场景:
  安静展示: 风扇 0% + 呼吸灯(青) + 不发声
  热血体验: 风扇 100% + 流星灯(红) + 引擎声(高)
  优雅巡航: 风扇 40% + 渐变灯(金) + 引擎声(低)
  自定义:   用户保存当前所有参数到 NVS

APP 端一键切换 + 固件端 O(1) 参数加载
```

### P9: 多设备架构预留

**不现在做**，但架构要预留扩展：
- `ble_connection_provider` 当前管理单个设备，State 改为 `List<ConnectedDevice>` 即可支持多设备
- 协议中设备 ID 用 `MAC` 前缀区分：`DEV1:FAN:50\n`
