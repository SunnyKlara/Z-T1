# ⚠️ 多会话协作协议

> **优先级**: CRITICAL — 本文件是防止项目腐烂的最后防线。
> **作用域**: 全局 — 固件端和 APP 端对话都必须遵守。
> **铁律**: 任何对话中，本文件中的规则不可被"方便"或"快速"等借口覆盖。

---

## 一、协作场景

```
你同时打开多个 Kiro 对话窗口：

  对话A ─── 固件开发 (ESP32 C代码, Bug修复, 新功能)
  对话B ─── APP 开发 (Flutter Dart代码, UI面板, BLE层)
  对话C ─── 架构讨论 (steering文档, 设计决策)

每个对话的 AI 共享同一份文件系统，但各自独立运行。
```

---

## 二、冲突风险清单

| 风险 | 严重程度 | 描述 |
|------|---------|------|
| **协议契约冲突** | 🔴 致命 | 两端对同一命令理解不同 |
| **LED预设不一致** | 🔴 致命 | 固件14种颜色与APP14种颜色不同 |
| **steering 文件并发修改** | 🟡 严重 | 两端同时编辑同一个 steering 文件 |
| **命名不一致** | 🟡 严重 | 固件用 `setLED`，APP用 `setLed` |
| **状态管理冲突** | 🟡 严重 | 固件的 AppState 字段改了，APP 不知道 |

## 三、核心铁律

### 铁律 1: 协议契约是唯一真值源

```
protocol-contract.md = 软硬件之间的"合同"

任何 BLE 命令、响应格式、LED预设、速度映射的变更：
  → 必须先更新 protocol-contract.md
  → 再改代码
  → 顺序不能反
```

### 铁律 2: 硬件参数以 hardware-config.md 为准

```
hardware-config.md = 所有硬件参数的唯一来源

任何硬件参数变更（灯珠数量、引脚分配、频率等）：
  → 必须先更新 hardware-config.md
  → 记录变更原因和日期
  → 再改代码
  → 顺序不能反
```

### 铁律 3: 每段对话必须声明工作范围

```
对话的第一条消息中声明：
"当前对话工作范围: [固件端 / APP端 / 两者 / 文档]"
```

**工作范围对应的文件权限：**

| 范围 | 可修改 | 不可修改 | 修改后需同步 |
|------|--------|---------|------------|
| 固件端 | `firmware/**/*.c`, `*.h` | `zcritical/` | `protocol-contract.md` |
| APP端 | `zcritical/lib/**/*.dart` | `firmware/` | `protocol-contract.md` |
| 两者 | 全部 | — | 全部 |
| 文档 | `steering/**/*.md` | — | 全部 |

### 铁律 3: 协议变更必须标记为 PENDING

```
修改 protocol-contract.md 添加新命令时：
  ### 2.XX 新命令 (PENDING — 固件已实现, APP待实现)

当两端都实现后：
  ### 2.XX 新命令 (✅ 固件OK / APP OK)
```

## 四、⚠️ 硬约束机制

> **以下不是建议，是规则。违反 = 出错。**

### 硬约束 1: 提交前合规检查报告（每段对话必输出）

每次文件修改提交前，AI 必须输出此报告：

```
═══════════════════════════════════════════════════════════════
📋 合规检查报告

文件行数:
  [文件名] [行数]/[上限] [✅/❌]

架构边界:
  [x] presentation/ 未 import data/
  [x] drivers/ 未 import ui/
  [ ] ❌ 问题: [描述]

协议一致性:
  [x] LED 预设数量: 14 (与固件一致)
  [x] BLE 命令格式: 与 protocol-contract.md 一致

决策:
  [放行 / 需要修复后重新提交]
═══════════════════════════════════════════════════════════════
```

### 硬约束 2: 每文件头嵌入约束声明

每个被修改的文件顶部，AI 必须确保存在（或补充）以下声明块：

```dart
// ══════════════════════════════════════════════════════════════
// STEER: 反臃肿 | max_lines=350 | 修改此文件前确认读过 anti-bloat.md
// ══════════════════════════════════════════════════════════════
```

**为什么：** 下一段对话的 AI 打开这个文件时，约束就在它眼前。它不能说自己不知道。

### 硬约束 3: 每次对话结束时更新 session-handoff

```
固件对话 → 更新 firmware/zcritical-esp/.kiro/session-handoff.md
APP对话   → 更新 zcritical/.kiro/session-handoff.md

记录: 完成了什么、留下了什么 TODO、下一个对话从哪开始。
```

### 硬约束 4: 新增文件必须有职责声明

```dart
/// 职责: [一句话说清楚这个文件做什么]
/// 不做什么: [明确排除的职责]
```

或 C 版本：

```c
/**
 * 职责: [一句话说清楚这个文件做什么]
 * 不做什么: [明确排除的职责]
 */
```

### 硬约束 5: 文件行数上限（不可协商）

| 区域 | 最大行数 |
|------|---------|
| APP 业务代码 (`zcritical/lib/`) | 350 行 |
| APP 配置文件 | 500 行 |
| 固件 `main.c` | 500 行 |
| 固件 `drivers/*.c` | 300 行 |
| 固件 `services/*.c` | 400 行 |
| 固件 `ui/*.c` | 350 行 |

**违反时**: 不能继续往该文件添加代码。必须先拆分再继续。

## 五、⚠️ 防 AI 偏离机制

### 问题：AI 在多轮对话后会偏离去向

```
对话第1轮:  严格遵守 steering，每文件 200 行
对话第10轮: "这里加 50 行方便"
对话第20轮: "先放着，后面再说"
对话第50轮: 文件 800 行，无人记得原来的约束
```

### 对策：每次对话开始时强制读取核心文件

*(见上文已更新列表)*

### 对策：问题-解决方案映射表

| 问题 | 解决方案 | 对应文件 |
|------|---------|---------|
| AI 说没问题但实机就是不行 | 对照已知坑位检查 | `known-pitfalls.md` |
| AI 做 UI 没有美感 | 使用设计令牌 + 验收清单 | `ui-design-tokens.md` + `ui-review-checklist.md` |
| AI 只从代码逻辑思考 | 遵守用户体验指南 | `ux-guidelines.md` |
| AI 改了参数但没同步 | 必须先改硬件配置文档 | `hardware-config.md` |
| AI 犯迷糊不认错 | 坚持技术合伙人定位 | `ai-proactive-guidance.md` |
| 协议格式不统一 | 协议契约是唯一真值源 | `protocol-contract.md` |
| 对话上下文丢失 | 更新 session handoff | `session-handoff.md` |

### 对策：文件头约束声明是 AI 的"护栏"

当 AI 打开一个文件时，第一行就看到约束声明：

```dart
// STEER: 反臃肿 | max_lines=350
```

这意味着 AI 不能在不知道约束的情况下修改文件。即使它没有读取独立的 steering 文件，文件内的约束字已经告诉了它规则。

## 六、冲突时的解决原则

### 协议理解冲突

```
以固件端为准（固件是硬件真值源）。

但必须写在 protocol-contract.md 中。
如果固件端实现有误 → 修改 protocol-contract.md → 两端同步。
```

### 代码冲突

```
固件改 C 文件、APP 改 Dart 文件 → 物理上不可能冲突。
因为它们在完全不同的目录下。
只有 protocol-contract.md 可能同时被改 → Git 冲突时手动合并。
```

---

## 七、对话结束时的标准收尾

每个对话结束时，AI 必须自动执行：

1. ✅ 输出合规检查报告
2. ✅ 更新对应域的 `session-handoff.md`
3. ✅ 如果有协议变更，确认 `protocol-contract.md` 已更新
4. ✅ 提交代码（如适用）
