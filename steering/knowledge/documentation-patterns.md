# AI 协作操作系统 — 可复用实践全面提炼

> **来源**: ZCritical T1 项目（ESP32-S3 固件 + Flutter APP，15+ 场 AI 对话）
> **用途**: 从实战中提炼出的完整 AI 协作体系，覆盖知识管理、协作模式、质量保障、自动化四大维度
> **创建**: 2026-05-12 | **修订**: 2026-05-12（从"文档管理"扩展为"完整操作系统"）

---

## 一、为什么需要这套体系

传统软件项目的文档服务于"人与人"的协作。AI 协作项目有三个根本不同：

| 传统项目 | AI 协作项目 |
|---------|-----------|
| 团队成员有持续记忆 | AI 每次对话从零开始 |
| 口头传递隐性知识 | 隐性知识必须显式化 |
| 文档是参考资料 | 文档是 AI 的"大脑加载项" |
| 文档过时影响效率 | 文档过时直接导致 AI 产出错误代码 |

**核心洞察**：在 AI 协作中，文档不是"写给人看的参考"，而是"AI 的操作系统"。文档质量直接决定 AI 产出质量。

---

## 二、七个可复用模式

### 模式 1：唯一入口 + 2 分钟启动

**问题**：新 AI 对话面对 20+ 个文档，不知道从哪开始读。读多了浪费算力，读少了缺乏上下文。

**解决方案**：创建一个唯一入口文件（START-HERE.md），满足以下约束：
- 2 分钟内读完（≤50 行）
- 包含项目一句话定位
- 包含 AI 能力边界声明
- 包含当前状态指针（指向 session-handoff）
- 包含"深入阅读"索引（按需读取，不强制）

**模板**：

```markdown
# 启动入口
## 项目
[一句话描述产品 + 技术栈]

## AI 能力
能做什么 / 不能做什么（写死，不要让 AI 每次猜）

## 当前状态
→ [指向 session-handoff 文件路径]

## 必知规则（5 条以内）
1. ...
2. ...

## 深入（按需读取）
- [主题] → [文件路径]
```

**适用场景**：任何使用 AI 辅助开发的项目，无论规模。

**为什么有效**：消除了 AI 的"冷启动焦虑"。AI 不需要猜测该读什么，入口文件告诉它一切。

---

### 模式 2：Session Handoff — 对话间状态传递

**问题**：AI 对话有上下文窗口限制，长项目必须跨多次对话。每次新对话，AI 不知道上次做到哪、什么已完成、什么待做。

**解决方案**：维护一个 session-handoff.md 文件，每次对话结束时更新。

**必须包含的信息**：
- 当前阶段标识（一眼看出进度）
- 已完成的工作清单（带验证状态）
- 下一步待做事项
- 关键决策记录（本次对话做出的选择）
- 编译/构建状态（最后一次验证结果）

**模板**：

```markdown
# [模块名] — 会话交接

## 当前阶段：[阶段名] → [下一阶段]
[一句话描述当前状态]

## 已完成
| 任务 | 状态 | 验证 |
|------|------|------|
| ... | ✅ | idf.py build 零错误 |

## 本次对话决策
- 决策 1：选择了 X 方案，原因是 Y
- 决策 2：...

## 下一步
1. ...
2. ...

## 编译状态
[最后一次构建命令 + 结果]
```

**适用场景**：任何需要多次对话才能完成的开发任务。

**为什么有效**：把"对话记忆"外化为文件。新 AI 读完 handoff 就拥有上一个 AI 的全部上下文，不需要重新探索。

---

### 模式 3：唯一真值源（Single Source of Truth）

**问题**：同一个配置/参数在多处定义，修改时遗漏某处，导致系统不一致。在 AI 协作中更严重——AI 可能读到过时的副本。

**解决方案**：对每类关键信息指定唯一权威文件，其他地方只引用不复制。

**实践规则**：
1. 识别项目中的"不可变约束"（硬件参数、协议格式、API 契约等）
2. 每类约束只在一个文件中定义
3. 其他文件需要引用时，写文件路径而非复制内容
4. 如果发现副本存在，归档副本并标注权威源

**示例**：

| 信息类型 | 唯一真值源 | 禁止在其他地方定义 |
|---------|-----------|-----------------|
| 硬件引脚 | `hardware-config.md` | 代码中的 #define 必须与此文件一致 |
| API 协议 | `protocol-contract.md` | 前后端各自的协议文档归档 |
| 设计令牌 | `design-tokens.md` | 代码中的颜色/字号常量引用此文件 |

**适用场景**：任何前后端分离项目、硬件+软件项目、多团队协作项目。

**为什么有效**：消除了"改了一处忘了另一处"的系统性风险。AI 只需要读一个文件就能获得权威信息。

---

### 模式 4：知识传承文档 — 记录"为什么"而非"是什么"

**问题**：传统文档只记录"做什么"和"怎么做"，不记录"为什么这样做"。新人（或新 AI）看到规则但不理解背景，要么盲目遵守（僵化），要么随意违反（混乱）。

**解决方案**：为每个重要决策记录完整的决策链：

```
问题 → 争论过程 → 结论 → 违反后果
```

**文档类型**：

| 类型 | 内容 | 示例 |
|------|------|------|
| 教训录 | 从失败中提炼的规则 | "为什么不能一次性写完所有文件" |
| 失败分析 | 前项目/前方案为什么失败 | "Reference 项目为什么烂尾" |
| 坑位清单 | 已踩过的技术坑 + 解决方案 | "BLE MTU 分片导致命令截断" |

**每条教训的结构**：

```markdown
## 教训 N：[标题]
**问题**：发生了什么
**争论**：各方观点是什么
**结论**：最终决策是什么
**违反后果**：如果不遵守会怎样
```

**适用场景**：任何有迭代历史的项目，特别是从旧项目重构的项目。

**为什么有效**：新 AI 不只知道规则，还知道规则背后的血泪。它会更谨慎地对待这些规则，而不是在"看起来合理"的时候随意违反。

---

### 模式 5：AI 能力边界显式声明

**问题**：AI 反复询问已知信息（"你有硬件吗？""能运行测试吗？"），浪费对话轮次。或者 AI 尝试做它做不到的事（烧录固件、实机测试）。

**解决方案**：在入口文件中写死 AI 的能力边界表。

**模板**：

```markdown
## AI 能力边界

| 能力 | 状态 | 说明 |
|------|------|------|
| 读取项目代码 | ✅ | 所有源文件 |
| 运行构建命令 | ✅ | `npm run build` / `idf.py build` |
| 运行测试 | ✅ | `npm test` / `flutter test` |
| 部署到生产 | ❌ | 用户操作 |
| 实机硬件验证 | ❌ | 用户反馈结果 |
| 联网搜索文档 | ✅ | 可查官方文档 |
| 访问数据库 | ❌ | 需要用户提供查询结果 |
```

**关键规则**：
- "不要问有没有 X"——如果能力表说了 ❌，AI 直接跳过，不浪费一轮对话确认
- "用户操作"类能力——AI 产出代码/命令，用户执行并反馈结果

**适用场景**：所有 AI 辅助开发项目。

**为什么有效**：每场对话节省 1-3 轮无意义的确认。15 场对话下来，节省的算力相当于多写了几个完整模块。

---

### 模式 6：防腐烂纲领 — 让破坏规则的成本高于遵守

**问题**：代码质量规范写了没人遵守。AI 尤其容易"顺从"——用户说"先这样吧"，AI 就不再坚持。

**解决方案**：不是写"应该怎样"的规范，而是设计"违反时会被发现"的机制。

**七道防线模型**：

| 防线 | 机制 | 检测时机 |
|------|------|---------|
| 1. 质量信号 | 文件行数/复杂度作为预警信号（不是硬限制） | AI 写代码时自查 |
| 2. 架构边界 | 目录结构 = 依赖方向，违反即编译失败 | 编译时 |
| 3. 职责声明 | 每个文件顶部声明"做什么/不做什么" | Code Review |
| 4. 新功能三问 | 属于哪个域？文件在哪？能容纳吗？ | 动手前 |
| 5. 原子提交 | 一次提交只做一件事 | 提交时 |
| 6. 定期审计 | 月度检查超标文件 | 定期 |
| 7. 文档同步 | 改代码必须同步改文档 | 提交时 |

**核心理念**：
- 防线是"发现问题"的机制，不是"限制创造"的枷锁
- 职责单一 > 行数限制（500 行但职责清晰 > 5 个 100 行碎片）
- "先这样，后面再整理" = 永远不会整理

**适用场景**：任何需要长期维护的项目。

**为什么有效**：把质量保障从"人的自觉"变成"系统的约束"。AI 有了明确的自检清单，不需要人盯着。

---

### 模式 7：争议义务 — AI 必须主动提出担忧

**问题**：AI 默认顺从用户。用户说什么就做什么，有担忧也不说。结果用户的决策建立在不完整的信息上。

**解决方案**：在 AI 行为规范中写入"争议义务"条款。

**规则**：

```
用户提出方案 → AI 确认理解 → AI 说出担忧（如果有）→ 争论 → 共识/用户坚持 → 记录 → 执行
```

**AI 必须做的**：
1. 有担忧就说，不藏着
2. 说完担忧后，如果用户坚持，尊重决策并全力执行
3. 记录争论过程和最终决策

**AI 不能做的**：
- 用预设规则代替真正思考
- 用户坚持后继续纠缠
- 有担忧但不说（最危险的反模式）

**适用场景**：任何 AI 作为"技术合伙人"而非"代码打字员"的场景。

**为什么有效**：把 AI 从"执行工具"提升为"思考伙伴"。用户的决策质量因为有了 AI 的专业反馈而提高。

## 三、文档体系的分层架构

### 推荐的三层结构

```
项目根/
├── steering/                    ← 全局层：跨模块共享
│   ├── START-HERE.md           ← 唯一入口
│   ├── specs/                  ← 不可变约束（协议、硬件、API）
│   ├── guides/                 ← 操作指南（Git、命名、AI行为）
│   ├── knowledge/              ← 知识传承（教训、坑位、失败分析）
│   └── archived/               ← 已废弃文档（保留历史）
│
├── 模块A/.kiro/                ← 模块层：模块特定
│   ├── session-handoff.md      ← 当前状态
│   └── steering/
│       ├── specs/              ← 模块架构
│       ├── guides/             ← 模块工作流
│       └── knowledge/          ← 模块技术债务
│
└── 模块B/.kiro/                ← 另一个模块
    └── ...
```

### 分层原则

| 层级 | 放什么 | 不放什么 |
|------|--------|---------|
| 全局 | 跨模块共享的约束和知识 | 模块特定的实现细节 |
| 模块 | 模块架构、当前状态、技术债务 | 全局规则的副本 |

### 文件数量控制

| 项目规模 | 全局文档数 | 每模块文档数 | 总计 |
|---------|-----------|------------|------|
| 小型（1人+AI） | 5-8 | 3-5 | 8-13 |
| 中型（2-3人+AI） | 8-12 | 5-8 | 15-25 |
| 大型 | 12-15 | 8-10 | 按模块数线性增长 |

**铁律**：如果新 AI 需要读超过 5 个文件才能开始工作，说明文档体系需要瘦身。

## 四、文档生命周期管理

### 文档的四个阶段

```
创建 → 活跃 → 过时 → 归档
```

| 阶段 | 特征 | 处理 |
|------|------|------|
| 创建 | 新写的文档 | 必须在索引中注册 |
| 活跃 | 最近 3 次对话中被引用 | 正常维护 |
| 过时 | 内容与代码不一致 | 立即更新或归档 |
| 归档 | 不再适用但有历史价值 | 移入 archived/ + 标注原因 |

### 归档标准

满足以下任一条件即归档：
- 被新文档完全替代
- 描述的架构/方案已废弃
- 连续 5 次对话未被引用且无人主动查阅

### 归档格式

```markdown
> ⚠️ 已归档 | 日期: YYYY-MM-DD | 原因: [原因] | 替代: [新文件路径]
```

## 五、配置双向同步规则

**核心原则**：代码和文档是同一个事实的两种表达。改一个必须改另一个。

| 触发 | 动作 |
|------|------|
| 改了代码中的配置值 | 同步更新对应的 spec 文档 |
| 改了文档中的规范 | 检查代码是否需要同步修改 |
| 新增了文件 | 更新架构文档 + session-handoff |
| 删除了文件 | 更新架构文档 + session-handoff |
| 做出了设计决策 | 记录到 session-handoff 或 knowledge/ |

**检测机制**：每次对话结束时，AI 自检：
- [ ] 本次改了代码吗？→ 文档是否同步？
- [ ] 本次改了文档吗？→ 代码是否需要同步？
- [ ] session-handoff 是否反映当前状态？

## 六、反模式清单

| 反模式 | 症状 | 正确做法 |
|--------|------|---------|
| 文档爆炸 | 20+ 文件，大量重叠 | 合并同类，每主题一个文件 |
| 规划当现实 | 文档描述未来架构，代码还是骨架 | 分离"当前状态"和"目标规划" |
| 规则矛盾 | 同一规则在不同文件中数字不同 | 唯一真值源，其他引用 |
| 过度仪式 | 改一行代码要更新 5 个文档 | 精简文档数量，合并同类 |
| 文档考古 | 归档文件没有标注原因和替代 | 归档时必须写原因 + 替代路径 |
| AI 全读 | 每次对话读完所有文档 | 入口文件 + 按需读取 |
| 只写不删 | 文档只增不减 | 定期审计，过时即归档 |

## 七、快速启动清单（新项目适用）

如果你要在一个新项目中应用这套体系，按以下顺序创建：

```
第 1 天：
  □ steering/START-HERE.md（唯一入口，50 行以内）
  □ [模块]/.kiro/session-handoff.md（当前状态）

第 1 周：
  □ steering/specs/ 下放 1-2 个不可变约束文件（API 契约、硬件参数等）
  □ steering/guides/ 下放 1 个 AI 行为规范

第 1 月：
  □ steering/knowledge/ 下开始记录教训和坑位
  □ 检查是否有文档重叠，合并

持续：
  □ 每次对话结束更新 session-handoff
  □ 每月审计一次文档健康度
```

**铁律**：先写代码，遇到问题再写文档。不要在第一天就创建 17 个文件。

## 八、度量指标

如何判断文档体系是否健康：

| 指标 | 健康 | 不健康 |
|------|------|--------|
| AI 冷启动时间 | ≤ 2 个文件即可开始工作 | 需要读 5+ 个文件 |
| 文档引用率 | 每个活跃文档最近 3 次对话被引用 | 有文件 1 个月没人看 |
| 规则一致性 | 同一规则只在一处定义 | 同一规则在 3 处定义且数字不同 |
| 文档/代码比 | 文档行数 < 代码行数的 20% | 文档比代码还多 |
| 新 AI 首次产出质量 | 读完入口+handoff 就能产出正确代码 | 需要多轮纠正 |

## 子系统二：AI 角色定义与协作模式

### 模式 8：AI 角色光谱 — 从打字员到技术合伙人

**问题**：大多数人把 AI 当"代码生成器"——说什么做什么。结果 AI 产出的代码用户看不懂、不敢改、不知道为什么这样写。

**解决方案**：显式定义 AI 的角色定位，写入 steering 文件。

**角色光谱**：

```
代码打字员 ←──────────────────────→ 技术合伙人
  │                                      │
  说什么做什么                          主动评估需求
  不提建议                              主动提风险
  写完就走                              写完给检查报告
  用户说停就停                          有担忧先说出来
```

**ZCritical 的选择**：技术合伙人。具体行为规范：
- 收到需求先评估合理性，不直接写代码
- 有担忧必须说出来（争议义务）
- 写完代码输出检查报告
- 主动带节奏——告诉用户下一步该做什么
- 用户坚持错误方案时：执行，但记录风险

**适用场景**：用户不是专业开发者但需要 AI 帮助构建复杂系统的场景。

**为什么有效**：把 AI 从"被动工具"变成"主动伙伴"。用户的决策质量因为有了专业反馈而提高。AI 不再只是执行指令，而是参与思考。

---

### 模式 9：契约驱动开发 — 先对齐再动手

**问题**：用户用自然语言描述需求，AI 直接写代码。中间有巨大的"理解鸿沟"——AI 做了大量隐式设计决策（状态放哪、错误怎么处理、文件怎么拆），用户看到代码才发现不对，但改起来代价很大。

**解决方案**：三层契约模型，在写代码前强制对齐。

```
第一层：功能契约（用户填模板描述需求）
  ↓ AI 确认理解
第二层：设计契约（AI 输出接口/模型/文件拆分方案，不写代码）
  ↓ 用户确认
第三层：实现（AI 按契约写代码）
```

**功能契约模板**：

```yaml
Feature: [功能名称]
Data:
  - [数据名]: [类型], [范围], [默认值]
Actions:
  - [操作名]: [输入] → [输出]
UI:
  - [UI 元素描述]
States:
  - [状态名]: [该状态下的表现]
```

**关键洞察**：问题发现在契约阶段成本极低（改一行文字），发现在代码阶段成本极高（改几百行代码）。

**适用场景**：任何需求不完全明确的功能开发。

---

### 模式 10：沟通模板 — 消除表达摩擦

**问题**：用户不知道怎么跟 AI 沟通。描述不清导致 AI 理解偏差，来回拉扯消耗大量对话轮次。

**解决方案**：为常见场景提供结构化沟通模板。

| 场景 | 模板结构 |
|------|---------|
| 提新需求 | 功能名 + 用户场景 + 操作路径 + 成功标准 |
| 反馈问题 | 实际现象 + 期望行为 + 复现步骤 |
| 讨论设计 | 主题 + 背景 + 不确定的地方 |
| 日常推进 | 当前状态 + 今天要做 + 约束 |

**AI 回应格式**（也标准化）：

```
【AI 分析】复述需求确认理解
【AI 方案】具体方案（不是代码）
【AI 建议】专业建议（必须有）
【下一步】建议下一步
```

**适用场景**：用户非技术背景，或团队中有多人与 AI 协作需要一致性。

**为什么有效**：把"怎么说"的认知负担从用户身上移除。用户只需要填空，不需要每次想措辞。

---

### 模式 11：产品思维注入 — AI 不只写代码

**问题**：AI 默认从"技术实现"角度思考。写出的代码功能正确但用户体验差——错误提示是技术术语、操作没有反馈、流程不符合用户心智模型。

**解决方案**：在 steering 中写入产品定位和 UX 原则，让 AI 在写每一行代码时都带着"用户会怎么感受"的视角。

**关键文档内容**：
- 产品定位（"体验型 App"而非"工具型 App"）
- 核心使用场景（5 个典型场景的完整描述）
- 交互设计原则（操作有手感、场景化 > 参数化）
- 错误提示规范（用户语言，不是技术术语）
- UX 检查清单（每个功能上线前必过）

**示例**：

| 技术事件 | ❌ 技术提示 | ✅ 用户提示 |
|---------|-----------|-----------|
| BLE 超时 | "Connection timeout" | "看来风洞有点远，靠近一点再试？" |
| CRC 校验失败 | "CRC32 mismatch" | "传输中出了点小问题，重新传一次" |

**适用场景**：任何面向终端用户的产品。

**为什么有效**：AI 不再只是"让代码跑通"，而是"让用户开心"。产品思维从文档渗透到每一行代码。

---

## 子系统三：质量保障体系

### 模式 12：架构边界即物理约束

**问题**：架构规范写了但没人遵守。AI 尤其容易违反——它不记得上次对话定的规则。

**解决方案**：把架构边界编码为目录结构 + 依赖方向规则，违反时编译失败或工具拦截。

**ZCritical 的实践**：

```
固件端：
  core/hal/    → 只依赖 ESP-IDF SDK
  core/state/  → 不依赖 modules/
  modules/     → 不互相依赖
  
APP 端：
  presentation/ → 只能 import domain/, core/
  data/         → 只能 import domain/, core/
  domain/       → 只能 import core/
  core/         → 不 import 任何业务代码
```

**关键**：不是靠"记住规则"，而是靠"目录结构本身就是约束"。文件放错目录 = import 方向错误 = 编译/lint 失败。

**适用场景**：任何有分层架构的项目。

---

### 模式 13：跨端契约同步 — 唯一真值源 + 变更流程

**问题**：前后端（或固件+APP）对同一协议的理解不一致，联调时才发现，浪费大量时间。

**解决方案**：

1. 协议定义只在一个文件中（`protocol-contract.md`）
2. 变更流程：先改文档 → 再改两端代码 → 联调验证
3. 禁止"先改代码再补文档"

**ZCritical 的协议契约包含**：
- BLE 连接参数（UUID、MTU、设备名）
- 完整命令表（31 条命令 + 参数范围 + 响应格式）
- 硬件主动上报事件
- LED 预设颜色表（14 种，两端 RGB 值必须对齐）
- 速度映射公式（两端必须一致）
- Logo 上传交互序列图
- 变更流程说明

**适用场景**：任何前后端分离项目、微服务间通信、硬件+软件协作。

---

### 模式 14：失败考古学 — 从旧项目提取教训

**问题**：重构项目容易重蹈覆辙。旧项目的失败原因如果不显式记录，新项目会犯同样的错。

**解决方案**：对旧项目做"尸检"，产出三类文档：

| 文档 | 内容 | 示例 |
|------|------|------|
| 失败分析 | 旧项目为什么烂尾 | "不是功能写不出来，是架构混乱导致维护成本不可承受" |
| 坑位清单 | 具体踩过的技术坑 | "BLE MTU 分片导致命令截断" |
| 正确用法 | 旧项目的正确打开方式 | "提取已验证参数 → 新架构重构逻辑" |

**ZCritical 的实践**：
- `why-reference-failed.md` — 6 个失败原因 + 对应的新项目应对
- `known-pitfalls.md` — 18 个坑位，每个有现象→根因→解决方案→验收标准
- `reference/README.md` — 旧代码的"矿藏地图"

**适用场景**：任何从旧项目重构/重写的项目。

---

### 模式 15：软硬件命名统一 — 消除概念混乱

**问题**：同一个概念在不同端用不同名字（固件叫 `strip`，APP 叫 `channel`），导致沟通和代码对接时反复确认。

**解决方案**：建立跨端命名对照表，写入 steering。

**表格结构**：

| 概念 | 中文 | 端 A 命名 | 端 B 命名 | 协议中的名字 |
|------|------|----------|----------|------------|
| ... | ... | ... | ... | ... |

**适用场景**：任何多端协作项目（前后端、移动端+服务端、硬件+软件）。

---

## 子系统四：自动化守护（Hooks）

### 模式 16：事件驱动的规则执行 — 让规范自动生效

**问题**：规范写了但依赖"AI 记得去遵守"。AI 不记得。每次新对话都可能忘记某条规则。

**解决方案**：用 IDE hooks 把关键规则绑定到事件上，自动触发提醒。

**ZCritical 的 7 个 Hook**：

| Hook | 触发事件 | 作用 |
|------|---------|------|
| 新对话启动 | promptSubmit | 强制读 START-HERE + session-handoff |
| 对话结束收尾 | agentStop | 强制更新 session-handoff |
| 架构边界守护 | 写 .c/.h 文件前 | 提醒检查依赖方向 |
| 协议契约守护 | protocol-contract.md 被修改 | 提醒同步两端代码 |
| 分支隔离 | 写代码文件前 | 提醒确认在正确分支 |
| 文档注册 | 新建 steering/*.md | 提醒在索引中注册 |
| UI 验收 | 写 Dart UI 文件前 | 提醒检查设计令牌和状态 |

**设计原则**：
- Hook 是"提醒"不是"阻断"——AI 看到提醒后自行判断
- 覆盖最容易犯错的环节（对话启动/结束、跨端同步、架构违反）
- 不过度——7 个 hook 覆盖 80% 的常见错误

**适用场景**：任何使用 AI IDE（如 Kiro、Cursor）的项目。

**为什么有效**：把"人记住规则"变成"系统提醒规则"。AI 的记忆不可靠，但 hook 的触发是确定性的。

---

### 模式 17：构建脚本封装 — 一键可重复

**问题**：AI 每次对话都需要知道怎么编译、怎么烧录、环境变量怎么设。如果构建步骤复杂，AI 容易出错。

**解决方案**：把所有构建步骤封装为一个脚本，AI 只需要调用一个命令。

**ZCritical 的实践**：`build_fw.ps1`
- 自动激活 ESP-IDF 环境
- 自动检查依赖文件（partitions.csv）
- 统一的参数接口（`build` / `flash` / `monitor` / `all`）
- 清晰的成功/失败输出

**适用场景**：任何构建步骤超过 2 步的项目。

---

## 子系统五：演化机制

### 模式 18：文档自我演化 — 从对话中生长

**问题**：文档体系不是一次性设计出来的。它需要随着项目推进不断演化。但如果没有机制，文档要么停滞不前，要么无序膨胀。

**ZCritical 的演化路径**：

```
第 1-3 场对话：只有 START-HERE + session-handoff（最小集）
第 4-6 场对话：加入 architecture + anti-bloat（遇到问题才加）
第 7-9 场对话：加入 conversation-lessons + ai-proactive-guidance（从争论中提炼）
第 10-12 场对话：加入 known-pitfalls + troubleshooting（从事故中学习）
第 13-15 场对话：瘦身——归档过时文档，合并重复内容
```

**演化规则**：
1. 遇到问题才加文档（不预设）
2. 每 5 场对话审计一次（删除/合并/归档）
3. 文档数量有上限（全局 ≤ 15，每模块 ≤ 8）
4. 新文档必须在索引中注册（hook 自动提醒）

**适用场景**：所有项目。

**为什么有效**：避免了两个极端——"第一天就写 20 个文档"（过度准备）和"从不写文档"（知识丢失）。文档随项目有机生长。

## 总结：四大子系统的协同

```
┌─────────────────────────────────────────────────────┐
│              AI 协作操作系统                          │
├─────────────┬──────────────┬───────────┬────────────┤
│ 知识管理     │ 协作模式      │ 质量保障   │ 自动化     │
├─────────────┼──────────────┼───────────┼────────────┤
│ 唯一入口     │ 角色定义      │ 架构边界   │ 启动 Hook  │
│ Session     │ 契约驱动      │ 跨端同步   │ 收尾 Hook  │
│ Handoff     │ 沟通模板      │ 失败考古   │ 边界 Hook  │
│ 唯一真值源   │ 产品思维注入  │ 命名统一   │ 协议 Hook  │
│ 知识传承     │ 争议义务      │ 防腐烂     │ 构建脚本   │
│ 能力边界     │              │           │ 文档注册   │
└─────────────┴──────────────┴───────────┴────────────┘
```

**一句话总结**：你建立的不是一套文档，而是一个让"无记忆的 AI"能够像"有经验的团队成员"一样工作的完整操作系统。

## 给其他项目的快速启动建议

### 第 1 天（30 分钟）

```
□ START-HERE.md（唯一入口）
□ session-handoff.md（当前状态）
□ 1 个 hook：新对话启动时读这两个文件
```

### 第 1 周（遇到问题时加）

```
□ 协议/API 契约文件（如果是多端项目）
□ AI 能力边界声明
□ 1 个 hook：对话结束时更新 handoff
```

### 第 1 月（从错误中学习）

```
□ 教训录（从对话争论中提炼）
□ 坑位清单（从 bug 中积累）
□ 架构文档（从代码中提炼）
□ 防腐烂规则（从膨胀中反思）
```

### 持续

```
□ 每 5 场对话审计文档健康度
□ 合并重复、归档过时、补充缺失
□ 文档总量控制在"新 AI 读 2 个文件就能开始工作"
```
