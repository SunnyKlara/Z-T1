# ZCritical 开发节奏 — 四圈迭代模型

> 创建: 2026-05-09 | 替代: archived/global-development-roadmap.md
> 核心理念: 每圈都是可体验的完整状态

## 为什么不用旧 Roadmap

旧 roadmap 按技术分层排顺序。问题: 做完 HAL 驱动用户看不到东西，固件和 APP 各自开发最后组装对不上，没有体验验证节点。

新规则: 每圈结束 = 可演示状态，体验验证通过才进入下一圈。

## 四圈总览

```
第一圈: 体验原型  ->  第二圈: 通信联通  ->  第三圈: 功能完善  ->  第四圈: 打磨发布
[UI全链路]            [BLE走通]           [全部功能]           [上市准备]
Mock数据              真实数据            每功能验证            细节打磨
```

---

## 第一圈: 体验原型 (1-2周)

目标: 做出完整交互原型，Mock 数据驱动。

**APP 端任务**:
1. 补齐所有页面 UI: 连接设备页、关于页、调试页(开发者模式)
2. 完善交互链路: 页面跳转、按钮反馈、加载状态、空状态
3. Mock 数据让页面"活的": 点下去有反应、数字会变
4. 你亲自验收: 不满意改 UI -> 满意锁定

**固件端任务**:
1. LCD 显示基础文字和状态信息
2. 编码器响应验证: 旋转 -> 日志输出 -> 确认硬件正常
3. LED 点亮: 基本色测试 -> 确认 WS2812B 正常
4. 风扇 PWM 最低速: 确认 MOS 管能驱动风扇

验收标准: APP 你可完整走一遍用户流程；固件硬件能单独验证。

---

## 第二圈: 通信联通 (1-2周)

目标: APP 和设备 BLE 走通。Mock 数据替换为 BLE 真实数据。

进入前需确认的 DR: DR-001(协议) 已确认 / DR-004(连接体验) 待讨论

**APP**: data层 BLE 服务 + 协议解析 + Provider 绑定真实数据
**固件**: GATT 服务 + 协议解析 + 状态机

联调验收: APP发FAN:50 -> 风扇转 -> 固件回报 -> APP 显示 50

---

## 第三圈: 功能完善 (2-4周)

目标: 全部功能实现，逐个验证。

进入前需确认的 DR: DR-002(音频)已确认 / DR-003(LVGL)已确认 / DR-005(Logo)待讨论 / DR-006(LED)待讨论 / DR-007(编码器)待讨论

功能: LED预设/风扇全模式/加湿器/编码器交互/LCD UI页面/Logo上传/音频引擎/WiFi投射

---

## 第四圈: 打磨发布 (1-2周)

目标: 打磨到上市水准。

进入前需确认的 DR: DR-008(OTA)/DR-009(看门狗)/DR-010(诊断)/DR-011(自检)

任务: OTA/看门狗/诊断/自检/动画微调/性能优化/数据统计/场景模式/文档/打包

---

## 迭代原则

1. 每圈结束必须体验验证，通过再进入下一圈
2. DR 决策只在进入对应圈前讨论，不提前生产
3. 不跨圈抢跑
4. 实机测试从第二圈开始
