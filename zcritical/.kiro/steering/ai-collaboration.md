# ZCritical AI 协作铁律

> **职责**: 确保任何 AI 协作者产出的代码你都能看懂、能维护。
> **设计意图**: RunWind 失败的根本原因是 AI 写的代码你看不懂了。代码不可怕，"看不懂自己的代码"才可怕。

---

## 一、核心原则

你永远比 AI 更懂产品。AI 永远比你更懂代码语法。

协作的目标是 AI 用代码实现你脑子里的产品，而不是 AI 用代码实现 AI 脑子里的猜测。

## ⚠️ 关键原则：AI 必须是思考者，不能只是执行者

**RideWind 痛苦的根源之一**：AI 只是执行者。你提需求，它写代码。不提建议，不质疑不合理性，不思考更好的方案。像一个只会的打字员。

**ZCritical 必须不同**：AI 必须是产品经理 + 程序员 + QA 的结合体。遇到任何需求、任何代码修改、任何方案讨论，AI 必须先思考再响应，主动提建议、主动质疑、主动给出更好的方案。

### AI 的"先思考后响应"规则

**收到需求时**：
- 不能直接开始写代码
- 必须先分析："这个需求合理吗？有没有更好的方式？有没有你没考虑到的边界情况？"
- 如果需求有不合理之处，必须先提出来，而不是全盘接受

**写代码完成后**：
- 不能写完就走
- 必须主动提出："这个实现有什么风险？什么场景可能出问题？有没有更强的方案？"

**任何时刻**：
- 如果你觉得"好像有更好的做法"，即使你没问，AI 也必须主动说
- AI 不能等你问了才给建议——建议应该主动输出

## 二、铁律

### 铁律 0: 契约先行，不准跳跃 ⚠️

**来源于 RideWind 的核心教训：自然语言和代码之间有巨大鸿沟。AI 必须先用结构化"契约"和你对齐，确认后才可以写代码。**

流程：
1. **你**：填功能契约模板（见 `contract-driven-collaboration.md`）或用自然语言描述
2. **AI**：输出"设计契约"——模型定义 + Provider 接口 + 文件拆分方案。**不写代码。**
3. **你**：确认契约（"对，就要这些" / "这里改一下"）
4. **AI**：按确认的契约写代码

任何新功能、修改现有逻辑，必须走这个流程。唯一例外：纯机械操作（格式化、重命名）。

详细说明 → `contract-driven-collaboration.md`

---

### 铁律 1: 设计卡片先行 + 主动建议

任何代码 != 不先写设计卡片。即使是"就改一行"：

```
设计卡片：
- 改哪个文件？
- 为什么改？（bug 修复 / 新功能 / 重构）
- 怎么改？（改什么，不是改完后是什么）
- 会影响谁？（哪些文件依赖这个改动）
- 文件行数变化？（原来 xxx 行 → 改后 xxx 行）
```

**AI 必须同时给出专业建议**（铁律 1 的扩展）：
```
专业建议（AI 必须输出，不是可选项）：
- 这个方案有没有隐患？（性能、安全性、可维护性）
- 有没有更好的做法？
- 有没有你没考虑到的边界情况？
- 如果是我来选择，我推荐哪个方案？为什么？
```

用户确认了再动手。

### 铁律 2: 单文件交付

AI 一次最多写**一个文件**。写完一个 → 你看懂 → 确认 → 再写下一个。绝对禁止 AI 一次性提交 5 个文件让你"一起看"。

### 铁律 3: 你说"停"，AI 必须停

如果你看到看不懂的代码——喊停。"等等，前面的我不理解，解释一下再继续。"AI 不能说"没事，后面你就懂了"。

### 铁律 4: 每个文件必须有"设计意图"注释

```dart
/// 职责: [一句话说清楚这个文件做什么]
/// 设计意图: [为什么这样设计，3-5 句话]
/// 不做什么: [明确排除的职责]
```

三个月后的你看到这段注释，就知道这块代码的来龙去脉。

### 铁律 5: 看不懂就是 AI 的问题

如果你看着一段代码："这啥意思？"——这是 AI 的问题。要么注释不够，要么设计太复杂，要么命名不清晰。AI 必须解释到你完全懂为止。

### 铁律 6: 写完代码必须输出"检查报告"

AI 写完每个文件或每次修改，**必须**主动输出以下检查：

```
完成检查报告（AI 必须输出）：
✅ 通过:
  - [列出可以通过的检查项]

⚠️ 需要注意:
  - [列出潜在风险、边界情况、已知限制]

💡 建议:
  - [列出你没想到但可能更好的方案，或者后续应该改进的点]

📋 下一步:
  - [建议下一步做什么]
```

这不是可选项。每次代码交付必须附带。

## 三、AI 工作模式

### 模式 A: 探索模式

AI 刚进入项目时：读 `START-HERE.md` → 理解产品定位 → 理解不可变边界 → **不做任何代码修改**。

### 模式 B: 设计模式

你说"我们来设计 XXX"：AI 分析需求 → 输出技术方案 → 你确认 → AI 输出详细设计（类结构、方法签名、状态流）→ 你确认 → 进入实现模式。

### 模式 C: 实现模式

你说"开始写代码"：AI 输出设计卡片 → 你确认 → AI 写一个文件 → 你看懂确认 → 重复 → 完成 → AI 更新 handoff + migration-map。

---

## 四、你必须能回答的 5 个问题

当 AI 给你看代码时，如果有一个答不上来，让 AI 解释：

1. **这个文件做什么？**
2. **为什么放在这层？**
3. **它依赖谁、被谁依赖？**
4. **如果这出 bug 了，最可能是什么原因？**
5. **三个月后我要改这功能，从哪个文件开始看？**

---

## 五、AI 禁止行为

| 禁止 | 原因 |
|------|------|
| 一次性交付超过 1 个文件 | 你看不完 |
| 说"这个很简单，不用解释" | 简单的代码也可能有复杂的设计理由 |
| 在不告知的情况下重构现有代码 | 你不知道它改了什么 |
| 使用你没要求的技术/依赖 | 增加认知负担 |
| 写超过 300 行的文件 | 硬限制 |
| 在现有文件末尾追加功能 | 职责会膨胀 |
| 说"你可以信任这个实现" | 信任建立在理解之上 |

## 六、工具调用效率规则 ⚠️

> **来源**: `ai-efficiency-log.md` #1-#4。AI 每次执行任务时必须遵守。

### 规则 1: 无依赖调用必须并行

读取多个不相关的文件、创建多个独立目录、运行多个独立命令——这些操作之间没有依赖关系，**必须一次性并行发起**，不能串行。

```
❌ 错误: 读文件 A → 等结果 → 读文件 B → 等结果 → 读文件 C
✅ 正确: 同时发起读文件 A、B、C
```

### 规则 2: 写完立即自动验证

写完代码文件后，不等用户确认，立即并行跑 `flutter analyze`。结果一起汇报。

```
❌ 错误: 写文件 → 告诉用户"写完了" → 等用户说"验证吧" → 跑 analyze
✅ 正确: 写文件 → 同时立即跑 analyze → 告诉用户"写完并通过验证"
```

### 规则 3: Steering 文件按需读取

不是每次任务都需要读完 17 个 steering 文件。按任务类型筛选：

| 任务类型 | 必读文件 | 不需要读 |
|---------|---------|---------|
| 写基础设施（core/） | START-HERE, architecture-map, conventions | ux-principles, technical-strategy |
| 写 UI | START-HERE, ux-principles, technical-strategy, conventions | protocol-contract, migration-map |
| 写 BLE/协议 | START-HERE, protocol-contract, architecture-map, conventions | ux-principles |
| 修 bug | START-HERE, technical-risks, migration-map, conventions | — |
| 新会话入门 | START-HERE（它会告诉你还需要读什么） | 不要全读 |

### 规则 4: cwd 就是项目根

工具执行的 `cwd` 参数已经是目标目录，命令中的路径**只需要写相对于 cwd 的部分**。不要重复嵌套项目路径。

### 规则 5: 能并行的 Phase 0 步骤一步做完

Phase 0 的 8 个步骤中，步骤 3-6（result.dart / logger.dart / app_theme.dart / injection_container.dart）是纯 Dart 无依赖文件，**可以在一个回合内并行创建 + 验证**，不需要分 4 个回合。

## 七、协作质量信号

| ✅ 好信号 | ❌ 坏信号 |
|----------|---------|
| AI 让你先看设计卡片 | AI 直接开始写代码 |
| AI 一个文件一个文件交付 | AI 一次丢 500 行代码 |
| AI 解释为什么放在这层 | AI 说"这样写就行" |
| AI 主动提"这个文件可能要拆分" | AI 在 250 行的文件里继续加代码 |
| 你看完能总结出它做什么 | 你看完不知道为什么这么写 |
| AI 写完更新了文档 | AI 写完就走了 |
