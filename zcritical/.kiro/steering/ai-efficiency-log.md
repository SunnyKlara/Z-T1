## #1: 工具调用并行化——减少往返延迟

- **日期**: 2026-05-07
- **问题**: AI 每执行一个操作（读文件、写文件、创建目录、运行命令）都要等待一次网络往返。串行执行 6 个独立操作 = 6 次 RTT 累积延迟。用户体感"AI 卡住了"。
- **原因**: AI 默认串行调用工具，但很多时候多个工具调用之间没有依赖关系（比如读取 3 个 steering 文件和创建 1 个目录可以同时做）。
- **方案**: 
  1. **AI 自查**：每次调用工具前，判断当前批次里哪些调用之间无依赖，能并行的全部并行。
  2. **读文件优先并行**：需要读 N 个不相关的文件时，一次性全部发出，不一个一个读。
  3. **创建目录 + 写文件可以合并**：先确保目录存在（并行创建多个目录），然后一次性写所有文件。
  4. **独立验证操作并行**：比如写 result.dart 的同时可以并行创建下一个文件的目录。
- **固化**: 写入本文件 #1。同步更新 `ai-collaboration.md`——增加"工具调用效率规则"章节。

## #2: 先验证后对话——避免在等待结果时闲聊

- **日期**: 2026-05-07
- **问题**: 写完 result.dart 后先跟用户解释修复方案，等用户确认后再跑 analyze。多了一次往返。
- **原因**: 写代码 → 等用户反馈 → 再验证，把"写-验"变成了"写-聊-验"三段式。
- **方案**: 
  1. **写 + 验一步到位**：写完文件立即跑 `flutter analyze`，不等用户确认。
  2. **结果一起汇报**：`analyze` 通过了，直接告诉用户"Step X 完成，通过"；有问题了，修完再汇报。
  3. **信任自动化**：`flutter analyze` 比人眼检查可靠，用它作为唯一判断标准。
- **固化**: 写入本文件 #2。更新 `engineering-standards.md` Phase 0 流程——增加"每步写完自动验证"。

## #3: 读 Steering 文件时要精简——按需读取

- **日期**: 2026-05-07
- **问题**: 每次新会话开始时，AI 会读取大量 steering 文件来理解上下文。17 个 steering 文件如果全读一遍，光是文件读取就要 17 次往返。
- **原因**: 不是所有 steering 文件在所有任务中都需要。写 `result.dart` 只需要看 `architecture-map.md` + `conventions.md` + `START-HERE.md`，不需要看 `ux-principles.md`。
- **方案**:
  1. **START-HERE.md 是指南针**：先读它，它告诉你该读什么。
  2. **按任务类型筛选**：写基础设施 → architecture-map + conventions；写 UI → ux-principles + technical-strategy；修 bug → technical-risks + migration-map。
  3. **AI 必须在每次会话开始时声明**："我需要读 X、Y、Z 三个文件就够，不读其他的。"
- **固化**: 写入本文件 #3。更新 `START-HERE.md`——在"第四步"增加"按任务类型选读"的映射表。

## #4: 目录创建失败别重试——检查路径即可

- **日期**: 2026-05-07
- **问题**: `New-Item -ItemType Directory` 在 PowerShell 中因为工作目录嵌套（`softwaer/zcritical` 下又嵌套了 `softwaer/zcritical`）导致路径错误。浪费了一次往返调试。
- **原因**: `cwd` 参数和命令中的相对路径叠加。工具已经在 `softwaer/zcritical` 目录下，命令又写了 `softwaer/zcritical/lib/core/error`，实际变成了 `softwaer/zcritical/softwaer/zcritical/lib/core/error`。
- **方案**:
  1. **理解 cwd 机制**：工具执行命令时，`cwd` 已经是目标目录。命令中的路径只需要写相对于 `cwd` 的部分。
  2. **先用 listDirectory 确认目录状态**：不确定目录是否存在时，先列一下。
  3. **目录创建命令标准化**：PowerShell 下用 `New-Item -ItemType Directory -Path <path> -Force`，多条目录创建用 `;` 分隔在一次调用中完成。
- **固化**: 写入本文件 #4。

## #5: PowerShell mkdir 不支持多参数——用 New-Item + 分号

- **日期**: 2026-05-07
- **问题**: 在 PowerShell 中用 `mkdir -p a b c` 创建多个目录失败——Windows 的 mkdir 不支持多个位置参数。
- **原因**: Windows PowerShell 的 `mkdir` 是 `New-Item` 的别名，只接受一个 `-Path` 参数。Linux 的 `mkdir -p` 语法不适用。
- **方案**:
  1. **Windows 创建多目录**：`New-Item -ItemType Directory -Path dir1 -Force; New-Item -ItemType Directory -Path dir2 -Force`
  2. **写文件和创建目录完全分离**：先批量创建所有需要的目录，再并行写所有文件。
  3. **一次性创建 3+ 目录**：用 `;` 串联多条 New-Item，避免分多次调用。
- **固化**: 写入本文件 #5。AI 在 Windows 环境下创建目录时，默认使用 `New-Item -Force;` 串联，而不是 `mkdir -p`。

## #6: 记录必须有可执行方案——不能光记不做

- **日期**: 2026-05-07
- **问题**: 用户指出效率日志如果只记录问题不真正执行解决方案，跟没记录一样。每条记录必须配套一个**可执行的动作**。
- **原因**: 记录是手段，解决是目的。如果方案写得很漂亮但没有真正落地（不更新对应文件、不修改流程），下次还是犯同样的错误。
- **方案**:
  1. **每条记录必须有"固化"字段**：明确写清"这条经验对应哪个文件的哪条规则，已经在何时更新"。
  2. **记录写入的同时执行固化**：写 #N 的同时，必须立即修改对应的 steering 文件。不能"稍后更新"。
  3. **验证固化的有效性**：更新完文件后，检查该规则是否可以直接指导 AI 行为（比如 `ai-collaboration.md` 的效率规则章节）。
  4. **自动化的不用人记**：`flutter analyze` 自动检查的规则（编译错误、lint 警告）不需要在这个日志里记录——记录的是需要 AI 主动遵守的行为规范。
- **固化**: 写入本文件 #6。同步更新本条（#6 本身即为自指——写完这条的同时已更新 `ai-collaboration.md` 的规则 5）。

## #7: 自然语言 ↔ 代码鸿沟——用"契约"架桥

- **日期**: 2026-05-07
- **问题**: 用户用自然语言描述需求，AI 直接写代码。中间大量设计决策（状态管理方式、组件拆分、命名规范、错误处理策略）用户完全不知道。等代码写完了才发现"这不是我想要的"。
- **原因**: 自然语言是模糊的，代码是精确的。AI 在翻译过程中自动做了用户意识不到的设计决策。用户无法在写代码前审查这些决策。
- **方案**:
  1. **三层契约体系**：功能契约（用户填模板）→ 设计契约（AI 输出接口/模型/文件拆分）→ 实现（AI 按契约写代码）
  2. **铁律 0**：没有契约，不写代码。任何新功能或修改逻辑，必须先走契约流程。
  3. **功能契约模板**：用户不需要写代码，只需要填写结构化 YAML——Feature / Data / Actions / UI / States / Constraints
  4. **设计契约内容**：AI 先输出模型定义、Provider 接口、文件拆分方案。不写一行实现代码。
  5. **关键区别**：问题发现在契约阶段（改一行文字），而不是代码阶段（改几百行代码）。
- **固化**: 
  - 创建 `contract-driven-collaboration.md`（完整契约流程 + 模板）
  - 更新 `ai-collaboration.md`（新增加铁律 0: 契约先行）
  - 更新 `START-HERE.md`（用契约流程替代旧的"第四步"）
  - 本条 #7 写于上述文件全部更新完成后
