# ZCritical 工程标准

> **用途**: 定义从需求到发布的完整质量门禁体系。按大厂标准适配个人开发者 + AI 协作场景。
> **设计记录**: 2026-05-07。大厂靠的不是人聪明，是靠"每个环节有标准卡着，不达标不让过"。
> **核心理念**: 标准不是建议，是门禁。每道门必须通过了才能进入下一道。

---

## 零、质量门禁总览

```
需求 ──→ 设计 ──→ 开发 ──→ 测试 ──→ 发布 ──→ 运维
  │        │        │        │        │        │
  ▼        ▼        ▼        ▼        ▼        ▼
需求文档  技术方案  Code    测试报告  发布清单  监控告警
必须写   必须评审  Review  必须通过  必须检查  必须配置
```

**每一道门都有一个"通行证"。没有通行证，不能进入下一阶段。**

---

## 一、需求标准

### 1.1 需求文档必须写（大厂叫 PRD — 产品需求文档）

大厂里，产品经理不是口头说一句"加个 Logo 上传"就完了。他们要写 PRD。你也需要。

```
【需求文档模板】

功能名称: [一句话]

用户场景: 什么时候、什么情况下、用户想做什么？

前置条件: 用户需要先做什么才能用这个功能？

操作流程: 从开始到结束的每一步
  1. [第一步]
  2. [第二步]

成功标准: 什么算是"做好了"？（可验证，不是感觉）
  - 例如：选择到完成 ≤ 2 分钟
  - 例如：上传成功率 ≥ 95%

异常情况: 出错了会怎样？
  - 情况 1: [描述] → [预期行为]

不属于本功能: 明确不做什么（防范围蔓延）
```

### 1.2 需求优先级

| 级别 | 定义 | 示例 |
|------|------|------|
| P0 | 没有它产品没法用 | BLE 连接 |
| P1 | 核心竞争力 | Logo 上传、场景模式 |
| P2 | 锦上添花 | 3D 展示、陀螺仪联动 |
| P3 | 未来迭代 | 社区、云端同步 |

**规则：任何时候只做 1 个 P0/P1 需求。P2/P3 排队。**

### 1.3 需求评审（大厂叫 PRD Review）

- [ ] 需求文档写完整了吗？
- [ ] 优先级定了吗？
- [ ] 有"不属于本功能"的明确排除吗？
- [ ] 和硬件端有关吗？有关的话契约定了吗？

## 二、设计标准

### 2.1 技术方案必须写（大厂叫 Tech Design / 概要设计）

```
【技术方案模板】

功能名称: [同需求文档]

架构层级:
  UI:       presentation/screens/[xxx]/[xxx].dart
  状态:     presentation/providers/[xxx]/[xxx_provider].dart
  逻辑:     domain/usecases/[xxx].dart
  数据:     data/datasources/[xxx]/[xxx].dart

文件预估: 新增 M 个文件，修改 N 个，每个 ≤ 300 行

依赖关系: [Provider] → [UseCase] → [Repository] → [DataSource]

协议影响:
  新增命令: [如果有]
  修改命令: [如果有]
  无影响: [如果没有]

风险预估:
  - 风险 1: [描述] → [缓解措施]
```

### 2.2 设计评审（大厂叫 Design Review）

| 检查项 | 问题 |
|--------|------|
| 文件放对层了吗？ | presentation 不能直接 import data |
| 文件 ≤ 300 行吗？ | 每个新文件预估行数 |
| 职责单一吗？ | 每个文件只做一件事 |
| 和现有功能冲突吗？ | 查 architecture-map.md |
| 协议变更了吗？ | 变了 → 先更新 protocol-contract.md |
| 用户场景覆盖了吗？ | 成功 + 异常 + 边界 |

**有一项不通过，方案打回修改。**

## 三、开发标准

### 3.1 开发顺序（大厂叫"自底向上"）

```
1. 数据模型（domain/models/）── 先定义数据长什么样
        ↓ 编译通过
2. 数据源（data/datasources/）── 数据从哪来
        ↓ 编译通过
3. Repository 实现（data/repositories/）── 怎么取数据
        ↓ 编译通过 + 单元测试
4. UseCase（domain/usecases/）── 业务逻辑
        ↓ 编译通过 + 单元测试
5. Provider（presentation/providers/）── 状态管理
        ↓ 编译通过
6. Screen / Widget（presentation/screens/）── UI
        ↓ 编译通过 + 手动测试
```

**每一层写完必须编译通过才能进入下一层。** 不允许"先把所有骨架写好再一起编译"。

### 3.2 编码规范

参见 `conventions.md`。核心三条：

| 规则 | 执行方式 |
|------|---------|
| 文件 ≤ 300 行 | 硬限制，CI 失败 |
| import 方向正确 | 硬限制，CI 失败 |
| 顶部有职责声明 | 硬限制，PR Review 拒绝 |

### 3.3 Code Review

大厂里任何代码进入主干前必须有 Review。对于你 + AI 的模式：

**AI 写完每个文件后，你必须能回答 5 个问题：**
1. 这个文件做什么？
2. 为什么放在这层？
3. 它依赖谁、被谁依赖？
4. 如果这出 bug 了，最可能是什么原因？
5. 三个月后我要改这功能，从哪个文件开始看？

**如果你有一个答不上来，说明 AI 没写好——不是你没看懂。**

### 3.4 提交规范

参见 `development-workflow.md`。核心：

| ✅ | ❌ |
|----|----|
| `feat: 完成 Logo 数据包编码` | `update` |
| `fix: 修复 BLE 发送队列竞态` | `fix bug` |
| `refactor: 拆分 protocol_parser` | `refactor` |

**每次提交只做一件事。**

## 四、测试标准

### 4.1 测试金字塔

```
      ╱  E2E 测试 ╲      少量（实机联调）
     ╱   集成测试   ╲     适量（Repository + BLE mock）
    ╱    单元测试    ╲    大量（UseCase + 协议 + CRC32）
```

### 4.2 什么必须测试

| 测试对象 | 原因 | 工具 |
|---------|------|------|
| CRC32 算法 | 和固件一致，错一点全盘皆输 | 纯 Dart test |
| 协议解析器（50+ 命令） | RideWind 正则不一致导致误解析 | 纯 Dart test |
| UseCase 业务逻辑 | 核心逻辑必须有保障 | mocktail |
| Repository（mock BLE） | 数据层正确性 | mocktail |
| 发送队列逻辑 | RideWind 有竞态 bug | 纯 Dart test |

### 4.3 允许不测试

| 不测什么 | 原因 |
|---------|------|
| 简单 getter/setter | 语言保证 |
| UI 像素位置 | 视觉走查即可 |
| 第三方库内部 | 自带测试 |
| 简单的 Widget 组合 | 编译通过即保证 |

### 4.4 测试覆盖率目标

| 层 | 目标覆盖率 |
|----|----------|
| domain/（UseCase + models） | 100% |
| data/datasources/protocol/ | 100% |
| data/datasources/transmission/ | 80% |
| data/repositories/ | 80% |
| presentation/providers/ | 60% |

**domain 层 100% 不是目标，是底线。** domain 是纯 Dart，没有外部依赖，没有理由不测。

## 五、发布标准

### 5.1 发布前检查清单

- [ ] 所有 P0/P1 功能测试通过
- [ ] 手动测试清单全部通过（见 engineering-process.md）
- [ ] `flutter analyze` 零错误
- [ ] 所有单元测试通过
- [ ] 无超过 300 行的文件
- [ ] 无跨层 import
- [ ] 包名: `com.zcritical.ridewind`
- [ ] 版本号已更新
- [ ] session-handoff.md 已更新

### 5.2 发布流程

```
内部测试（你自己）→ 封闭测试（找几个人）→ 开放测试 → 正式发布
     1-2 天              3-5 天             3-5 天       发布
```

**不要跳过封闭测试。** 找 2-3 个朋友，把 App 给他们用，坐旁边看他们操作。你会发现 10 个自己测试发现不了的问题。

### 5.3 回滚准备

每次发布必须有回滚方案：
- 上一个稳定版本可随时重新发布
- 数据库/本地存储向后兼容

## 六、运维标准

### 6.1 日志规范

```dart
// ✅ 好的日志
logger.i('BLE 连接状态变更: disconnected → connecting');
logger.w('Logo 上传第 $seq 包重试第 $retry 次');
logger.e('CRC32 校验失败: 预期 $expected, 实际 $actual');

// ❌ 坏的日志
print('error');
debugPrint('data: $data');
// 没有上下文，出了问题不知道在哪
```

### 6.2 版本号规则

```
1.0.0+1

主版本.次版本.修订号+构建号

主版本: 大重构、不兼容的 API 变更
次版本: 新功能
修订号: Bug 修复
构建号: 每次构建递增
```

### 6.3 崩溃率目标

| 指标 | 目标 |
|------|------|
| 崩溃率 | < 0.1%（每 1000 次启动不超过 1 次崩溃） |
| ANR 率 | < 0.5%（Android 无响应） |

## 七、文档标准

### 7.1 文档必须和代码同步更新

| 代码变更 | 需更新的文档 |
|----------|------------|
| 新增/修改协议命令 | `protocol-contract.md` |
| 新增/拆分文件 | `architecture-map.md` + `migration-map.md` |
| 发现新风险 | `technical-risks.md` |
| 新编码规范 | `conventions.md` |
| 阶段完成 | `reconstruction-blueprint.md` + `session-handoff.md` |

**不更新文档的代码 = 没完成。**

### 7.2 Code 即文档

每个文件顶部必须有：

```dart
/// 职责：[一句话]
/// 设计意图：[为什么这样设计]
/// 不做什么：[明确排除的职责]
```

## 八、质量门禁速查表

任何一个 `❌` 都不能进入下一阶段。

### 需求 → 设计 通行证

- [ ] 需求文档写完整了
- [ ] 优先级标注了
- [ ] AI 给出了专业分析
- [ ] 你和 AI 对需求的理解一致

- [ ] 技术方案写完整了

- [ ] import 方向正确

- [ ] 每一层编译通过
- [ ] `flutter analyze` 零错误
- [ ] 每个文件有职责声明
- [ ] import 方向检查通过

- [ ] 所有单元测试通过

- [ ] 发布检查清单全部勾选

## 九、大厂 vs 你的项目：差异与适配

| 大厂做法 | ZCritical 适配 |
|---------|---------------|
| 产品经理写 PRD | 你写需求文档（按模板填） |
| 技术负责人写 Tech Design | AI 写技术方案，你确认 |
| 同事 Code Review | AI 的自检 + 你回答 5 个问题 |
| QA 团队测试 | 自动化测试 + 你的手动测试 |
| SRE 运维监控 | logger 日志 + 崩溃率监控 |
| 项目管理排期 | 优先级 + 一次只做一个 P0/P1 |
| CI/CD 流水线 | flutter analyze + flutter build |

## 十、如果只能记住一条

> **每一阶段都有通行证。没有通行证，不进下一阶段。**
>
> 需求没写清楚 → 不动手设计。
> 设计没评审 → 不动手写代码。
> 代码没 Review → 不合并到 main。
> 测试没通过 → 不发布。

RideWind 的问题不是某一行代码写错了。是**每一道门都没关——需求随口一说就开始写、代码写完就合并、从不测试、从不 Review。**

每一道门单独看都不费时间（写个文档 10 分钟、跑个测试 1 分钟），但合在一起，决定了最终产品的质量。
