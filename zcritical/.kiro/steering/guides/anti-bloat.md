# ⚠️ ZCritical 防臃肿纲领

> **优先级**: CRITICAL — 违反此纲领 = 代码腐烂
> **作用域**: APP端 — 所有 zcritical/lib/ 下的 Dart 文件

> **用途**: 确保重构后的代码不会在几个月内再次腐烂。
> **核心理念**: 臃肿不是一天形成的，是在每一次"加50行就行"的妥协中累积的。
>             防臃肿靠的不是"写好代码"，而是**让破坏规则的成本高于遵守规则的成本**。

---

## 一、理解臃肿是怎么发生的

```
第1天:  干净架构，职责清晰
第30天: 加个新功能，往已有文件塞 — "反正这个文件已经处理LED了"
第60天: 修个bug，加个兜底逻辑 — "这里特殊处理一下就行"
第90天: 赶进度，复制粘贴 —"这个逻辑跟那边差不多，改几个参数就行"
第120天: 文件职责模糊了 —"算了，能跑就行，后面再说"
第180天: 文件没人敢动了 —"这谁写的？"

问题出在第30天，不是第180天。
每一次妥协在当时都是合理的，但没有人阻止它发生。
```

## 二、七道防线

### 防线 1: 职责单一优先（核心防线）

**核心原则**: 一个文件只做一件事。职责清晰是唯一标准。

**判断标准**（按优先级）：
1. 能用一句话说清楚这个文件做什么吗？→ 职责单一
2. 新加的功能属于"不做什么"范围吗？→ 越界了，该拆
3. 这个文件和同目录其他文件有重叠职责吗？→ 边界不清

**职责评估**：
- 职责模糊 → AI **主动评估**：是否需要拆分？
- 职责越界 → 必须拆分，**AI 必须说明理由**
- 职责混乱 → 强制拆分，没有例外

**行数指导**（非硬性限制，仅作预警参考）：
- < 200 行：正常范围
- 200-400 行：预警区，检查职责是否仍然单一
- > 400 行：必须说明理由（文件顶部注释），下次对话优先评估拆分

**例外通道**：
当文件确实需要超过 400 行时：
1. 在文件顶部说明理由（`/// 为什么超过400行: xxx`）
2. 确保职责仍然单一（不能因为长就塞多个职责）
3. 下次对话时优先评估是否可以拆分

**禁止**：在文件顶部标注 `ref_lines=XXX`。职责是目标，不是行数。

### 防线 2: 目录结构就是架构边界

**允许的 import 方向**（违反则 CI 失败）:

```
presentation/  → 只能 import domain/, core/, presentation/
data/          → 只能 import domain/, core/, data/
domain/        → 只能 import core/, domain/
core/          → 不能 import 任何业务代码（domain/data/presentation）
```

**检查工具**: `import_sorter` + 自定义 lint 规则，或 Dart 的 `layer_lint` 包。

**这意味着**:
- Screen 不能 import BLE Service → 强制通过 Repository 抽象
- Provider 不能 import UI Widget → 状态和UI彻底分离
- data 层不能 import presentation → 数据层不依赖UI

### 防线 3: 每个文件顶部有职责声明

```dart
/// 职责：[一句话说清楚这个文件做什么]
/// 不做什么：[明确排除的职责，防止越界]
```

示例：
```dart
/// 职责：BLE设备扫描，过滤、排序扫描结果
/// 不做什么：不管理连接、不解析协议、不存储设备信息
class BleScanner { ... }
```

**实践**: 当你想加新功能时，先看目标文件的职责声明：
- 如果新功能属于这个职责 → 可以加（但要检查职责是否仍然单一）
- 如果不属于 → 新建文件，永远不要往"不做什么"的范围内加代码

### 防线 4: 新功能三步走

每次加新功能，必须先回答三个问题，写在 commit message 里:

```
Q1: 这个功能属于哪个域？
    (BLE / LED / Logo / WiFi音频 / 设备状态 / 设置 / ...)

Q2: 这个域对应的文件在哪？
    (列出具体文件路径)

Q3: 现有文件能容纳吗？
    - 职责匹配吗？（看职责声明）
    - 加了之后职责仍然单一吗？
    → 如果任何一个答案是"否"，新建文件
```

### 防线 5: 一次提交只做一件事

| ✅ 允许 | ❌ 禁止 |
|--------|--------|
| 修一个 bug | 修 bug + 顺便重构 |
| 加一个新功能 | 加功能 + 顺便优化旧代码 |
| 拆分一个大文件 | 拆分 + 顺便改功能逻辑 |
| 重命名/重构 | 重构 + 顺便加新功能 |

**原因**: "顺便改"最危险。两个变更混在一起，review 看不出问题，回滚也回不干净。

### 防线 6: 每月瘦身审计

每月执行一次，用脚本自动化：

```bash
# 检查超过预警线的文件
find lib/ -name "*.dart" -exec wc -l {} + | sort -rn | head -20

# 检查跨层 import
grep -r "import.*presentation" lib/data/
grep -r "import.*data" lib/presentation/

# 检查是否有职责声明
grep -L "/// 职责：" lib/**/*.dart
```

### 防线 7: 代码审查清单

每次 PR 前自查：

| 检查项 | 状态 | 处理 |
|--------|------|------|
| 文件职责声明清晰 | ✅ 通过 | 继续 |
| 职责模糊 | 🟡 预警 | AI 自查 + 计划拆分 |
| 新增文件没有职责声明 | 🟡 预警 | 补上再合 |
| PR 中混了 bugfix + 功能 | 🟡 预警 | 拒绝，拆分提交 |
| Provider 同时管理 3 种以上状态 | 🟡 预警 | 讨论是否需要拆分 |
| import 跨层 | 🔴 严重 | 必须修复才能合 |
| 两天内没有人看 steering 文件 | 🔴 严重 | 说明文档已经过时 |
| "先这样，后面再整理" | 🔴 严重 | 后面永远不会整理 |

## 三、原型阶段与正式阶段 ⚠️

> **这条规则是 2026-05-08 从教训中学到的。**

### 核心问题

`home_page_view.dart` 最初被设计为"临时占位原型"，在文件头部明确写着：

```dart
// - 4 页均为假数据占位内容，用于体验布局比例和滑动手感。
```

但因为没有规定"原型何时退出"，代码在 766 行时仍然保持着原型身份，继续被往里面塞功能。

### 两阶段定义

| 阶段 | 特征 | 规范要求 |
|------|------|---------|
| **原型阶段** | 文件头部注明 `// 原型：...` 且有明确退出条件 | 可超行数指导线，但必须注明原因和退出时间 |
| **正式阶段** | 文件头部只有 `/// 职责：...` 职责声明 | 必须遵守所有规范（职责单一、架构边界等） |

### 退出条件：原型阶段结束的信号

原型文件必须在以下任一条件触发时，立即转为正式代码并拆分：

| 触发条件 | 处理方式 |
|---------|---------|
| 开始添加真实业务逻辑（不是假数据） | 立即拆分，不等 |
| 原型设计被用于真实功能 | 必须拆分后才能继续开发 |
| 开始修改/迭代该文件的功能 | 必须先拆分再改 |
| 进入下一个 Phase 阶段 | 上一 Phase 的所有原型必须退出 |

### 新建文件的原则

**永远不要以"原型"身份创建新文件。**

- 新建文件 → 立即遵守规范 → 写 `/// 职责：...`
- 如果不确定需求 → 用注释写 `// TODO: 正式实现前拆分` + 标注退出条件
- 禁止以"先占位"为由创建不符合规范的文件

### 规范漏洞总结

本次教训发现以下规范缺失，已补全：
- ✅ 新增：本节"原型阶段与正式阶段"
- ✅ 本对话中：明确了开发新功能的完整流程（见下方）

## 四、实验代码与正式代码同等规范 ⚠️

> **这条规则是 2026-05-08 从教训中学到的。**

### 核心原则

**"试试看"的代码和"正式开发"的代码必须遵守完全相同的规范。**

这不是可选的。这是从 RideWind 教训中学到的最重要的规则——RideWind 里有多少"先试试看"的代码，最后变成了没人敢动的死代码？

### 什么叫"试试看"

以下全部按正式开发流程对待：
- 在已有文件里加 `// TODO: 后面再改` 的代码
- 加了 `// 占位`、`// 临时`、`// 原型` 注释的代码
- 用户说"先看看效果"而写的任何 UI
- 任何不在 feature 分支上的实验代码

### 为什么"试试看"代码更危险

| 正式代码 | 试试看代码 |
|---------|-----------|
| 有职责声明 → 一眼看出做什么 | 无声明 → 三个月后看不懂 |
| 有设计契约 → 确认后才写 | 无契约 → 直接猜着写 |
| 遵循职责单一 → 不会无限膨胀 | "反正就几行" → 累积成灾 |
| 在 feature 分支 → 可以回滚 | 在 main 或随手改 → 污染主干 |

### "试试看"代码的处理方式

| 场景 | 正确做法 |
|------|---------|
| 不知道需求想清楚没有 | 用 `// TODO:` 注释标记，但同样遵守职责单一 |
| 想看效果验证设计 | 在 feature 分支实验，效果好合并，不好删分支 |
| 占了位置的代码 | **必须**写 `/// 职责：...`，哪怕是"待替换" |
| 不确定要不要保留 | 不写，直接问用户确认后再决定 |

### 规范触发点（必须立即执行）

- 职责开始模糊 → 立即拆分，不等
- 开始迭代/修改原型文件 → 必须先拆分
- 进入下一个 Phase → 上一 Phase 所有占位文件必须拆分或删除
- `// 占位`、`// TODO`、`// 原型` 注释超过 3 处 → 重构或删除

### 铁律

> **"先试试看"不是"不遵守规范"的借口。**
> 
> 如果你想实验——开分支，实验完了觉得好就合并（先拆分再合），觉得不好就删。
> 没有任何理由可以让"试试看"的代码绕过规范。

## 五、需要妥协时的原则

现实开发中总会遇到需要妥协的时候。当不得不妥协时，遵守优先级:

1. **架构方向不能妥协** — import 方向、层次边界
2. **职责单一不能妥协** — 一个文件只能做一个事
3. **行数可以临时超标** — 但必须开 issue，标注"下次迭代拆分"，不超过 400
4. **命名可以后期改** — 名字不对不破坏架构
5. **注释可以后期补** — 逻辑对最重要

**一句话**: 可以暂时允许乱，但必须让所有人知道这是乱的（issue + 职责声明标注），并且有明确的清理时间。
