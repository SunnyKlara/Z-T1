# ZCritical 工程管理规范

> **职责**: 定义从需求到上线的完整开发流程，按大厂标准适配个人开发者 + AI 协作场景。
> **设计意图**: RideWind 失败的根本原因之一是"Demo 驱动开发"——没有需求文档、没有设计评审、没有测试、没有运维。本规范解决"我不知道标准流程是什么"的问题。
> **不做什么**: 不讨论具体技术实现。

---

## 一、产品开发全流程（6 个阶段）

```
阶段1: 需求定义 → 阶段2: 方案设计 → 阶段3: 开发实现
阶段4: 测试验证 → 阶段5: 发布上线 → 阶段6: 运维迭代
```

每个阶段有明确的输入、输出、完成标准。不能跳过。

---

## 二、阶段 1: 需求定义

### 写需求文档（不是一句话描述）

```
功能名称: Logo 上传
用户场景: 在桌前，想把车队 Logo 传到风洞屏幕上
前置条件: 设备已连接
操作流程:
  1. 点击首页"定制"入口
  2. 选择本地图片
  3. 预览图片（240×240 裁剪确认）
  4. 选择目标槽位
  5. 点击上传
  6. 查看进度（百分比 + 预计时间）
  7. 完成后查看效果
异常情况:
  - 图片格式不支持 → "仅支持 PNG/JPG"
  - 图片太小 → "图片至少 240×240"
  - BLE 中断 → "连接中断" + 保留进度
  - 校验失败 → "传输异常" + 可重试
  - 槽位满 → 选择替换现有 Logo
成功标准:
  - 选择到完成 ≤ 2 分钟
  - 上传成功率 ≥ 95%
  - 任何步骤用户都知道发生了什么
```

### 优先级定义

| 级别 | 定义 | 示例 |
|------|------|------|
| P0 | 没有它产品没法用 | BLE 连接、风扇控制 |
| P1 | 核心竞争力 | Logo 上传、场景模式 |
| P2 | 锦上添花 | 陀螺仪联动、分享 |
| P3 | 未来迭代 | 社区、云端 |

## 三、阶段 2: 方案设计

### 技术方案模板

```
技术方案: [功能名]

架构层级:
  UI: presentation/screens/xxx.dart
  状态: presentation/providers/xxx/xxx_provider.dart
  逻辑: domain/usecases/xxx.dart
  数据: data/datasources/xxx/xxx.dart

依赖关系:
  xxx_provider → xxx_usecase → xxx_repository → xxx_datasource → ble_client

协议命令: (如果有新的)
  XXX:param:value

文件预计:
  新增 N 个文件，修改 0 个，均 ≤ 250 行
```

### 设计评审检查清单

| 检查项 | 问题 |
|--------|------|
| 架构合规 | 文件放对层了吗？ |
| 行数合规 | 每个新文件 ≤ 300 行？ |
| 职责单一 | 每文件只做一件事？ |
| import 合规 | 有没有跨层依赖？ |
| 可测试性 | 核心逻辑能 mock 测试吗？ |
| 用户体验 | 符合 ux-principles.md 吗？ |

## 四、阶段 3: 开发实现

### 开发顺序（自底向上）

```
1. domain/models/    数据模型（freezed）
2. core/             基础设施
3. data/datasources/ 数据源
4. data/repositories/ Repository 实现
5. domain/usecases/  业务逻辑
6. presentation/providers/ 状态管理
7. presentation/screens/   UI 界面
```

每写完一层，那一层独立可编译运行。

### 编码检查清单

- [ ] 文件顶部有职责声明
- [ ] 复杂逻辑有设计意图注释
- [ ] 文件 ≤ 300 行
- [ ] import 方向正确
- [ ] 公共 API 有文档注释
- [ ] 无硬编码字符串
- [ ] 无 `dynamic` 类型

### 提交规范

```
✅ "feat: 添加 Logo 上传会话状态机"
✅ "fix: 修复 BLE 发送队列竞态"  
✅ "refactor: 拆分 protocol_parser"

❌ "feat: Logo 上传 + 顺便修 bug + 优化 UI"
```

## 五、阶段 4: 测试验证

### 测试金字塔

```
      ╱  E2E 测试 ╲      少量
     ╱   集成测试   ╲     适量
    ╱    单元测试    ╲    大量
```

### 必须测试

| 测试什么 | 工具 |
|---------|------|
| 协议编解码（50+ 命令） | 纯 Dart test |
| CRC32（固件交叉验证） | 纯 Dart test |
| Repository（mock BLE） | mocktail |
| UseCase（逻辑分支） | mocktail |
| Provider（状态转换） | riverpod_test |

### 可以不测

| 不测什么 | 原因 |
|---------|------|
| 简单 getter/setter | 语言保证 |
| UI 像素布局 | 视觉走查即可 |
| 第三方库内部 | 自带测试 |

### 手动测试清单（发布前必过）

- [ ] 首次启动 → 权限 → 扫描 → 连接 → 完整走通
- [ ] 断连 → 自动重连 → 恢复控制
- [ ] 所有 14 种 LED 预设切换
- [ ] 风扇速度 0-100 发送正确
- [ ] Logo 上传 240×240 成功
- [ ] Logo 上传中途取消不破坏现有
- [ ] 深色/浅色模式切换
- [ ] 最小/最大屏幕显示正常
- [ ] 后台恢复 → 自动重连

## 七、阶段 6: 运维迭代

### 版本节奏

```
小版本 (1.0.x): 每周 — bug 修复
中版本 (1.x.0): 每月 — 新功能  
大版本 (x.0.0): 每季度 — 架构升级
```

### 迭代流程

```
收集反馈 → 分析优先级 → 需求文档 → 方案设计 → 开发 → 测试 → 发布 → 收集反馈
```

## 八、你 vs AI 分工

| 你负责 | AI 负责 |
|--------|---------|
| 需求定义（用户场景） | 技术方案设计 |
| 设计评审（确认方案） | 代码实现 |
| 测试决策（测什么） | 写测试代码 |
| 手动测试（真机） | 自动化测试 |
| 用户体验判断 | 代码质量保障 |

## 九、完整功能开发流程（以 Logo 上传为例）

```
[需求] 写 PRD → 用户场景 + 操作流程 + 异常 + 成功标准
  ↓
[设计] 写技术方案 → 文件位置 + 依赖 + 协议 + 行数预估
  ↓
[评审] 检查清单 → 架构 + 行数 + 职责 + import + UX
  ↓
[模型] domain/models/ → freezed 定义数据
  ↓ 编译通过
[数据] data/datasources/ → 数据源实现
  ↓ 编译通过 + 单元测试
[逻辑] domain/usecases/ → 业务逻辑
  ↓ 编译通过 + 单元测试
[状态] presentation/providers/ → 状态管理
  ↓ 编译通过
[UI] presentation/screens/ → 界面
  ↓ 编译通过 + 手动测试
[验收] 按 PRD 成功标准逐条验收
  ↓
[文档] 更新 migration-map + session-handoff
```

每走完一个 `↓`，你完全理解这一步的代码。
