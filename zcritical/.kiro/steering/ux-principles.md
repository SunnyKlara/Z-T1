# ZCritical 用户体验设计纲领

> **用途**: 重构的最高指导原则。当架构决策与用户体验冲突时，**用户体验优先**。
> **核心理念**: RideWind 能跑通但体验太差——这才是重构的真正原因。代码干净是为了让体验迭代更快，但干净不是目的。
> **北极星问题**: 用户用 ZCritical 操作风洞模型时，感觉到什么？

---

## ⚠️ 产品定位（必须理解）

| 维度 | 说明 |
|------|------|
| 产品 | **桌面风洞模型**（汽车风洞的 1:64 微缩版），桌面摆设品 |
| 目标用户 | 车模收藏爱好者、桌面潮玩玩家、STEM 科教产品消费者 |
| 核心体验 | **把玩、展示、个性化** — 不是"工具"，是"玩具" |
| 使用场景 | 坐在桌前悠闲操作，双手拿手机 |
| 核心情感 | 仪式感、工艺感、归属感 ("这是我的专属装备") |
| App 角色 | 驾驭风洞模型的精致控制终端 |

### 这意味着什么

- ❌ 不是: 工具型 App（快速、简单、用完离开）
- ✅ 是: 体验型 App（精致、有仪式感、愿意花时间把玩）
- ❌ 不是: 罗列参数和控制选项
- ✅ 是: 场景化操作 + 可选深入自定义
- ❌ 不是: 冷冰冰的工业界面
- ✅ 是: 有质感、有温度、值得拍照分享的界面
- ❌ 不是: "开车时单手盲操作"的设计
- ✅ 是: "坐在桌边双手细致把玩"的设计

## 一、RideWind 的用户体验问题清单（反面教材）

从 RideWind 中提取的用户体验缺陷，每个都必须在新设计中解决。RideWind 的很多问题是**产品定位错误**导致的——它用"工具型 App"的思路设计了"体验型产品"的界面。

### 连接体验
| 问题 | 用户感受 | zcritical 要求 |
|------|---------|---------------|
| 扫描→配对→连接流程冗长 | "怎么还连不上？把我的兴致都搞没了" | 打开 App → 自动找到设备 → 有仪式感的连接动画 |
| 连接状态不明确 | "到底连上了没？" | 连接过程是体验的一部分，不是枯燥的等待 |
| 断连无提示 | "怎么突然没反应了？" | 断连不破坏体验，优雅提示 + 自动恢复 |
| 多步骤手动操作 | "每次都要点好几下" | 默认自动连接上次设备，无需手动操作 |

### 控制体验
| 问题 | 用户感受 | zcritical 要求 |
|------|---------|---------------|
| 简单罗列参数 | "这就是个遥控器" | 场景化操作：一键切换到想要的效果 |
| 缺乏场景联动 | "每个都要单独调" | 预设场景模式，灯光+风速+音效联动 |
| 操作反馈不精致 | "感觉在操作工业设备" | 每一次交互都是微小的享受 |
| 缺少"把玩感" | "调完就没意思了" | 提供探索性——试试这个模式，看看效果 |

### 视觉/展示体验
| 问题 | 用户感受 | zcritical 要求 |
|------|---------|---------------|
| UI 没有质感 | "像个学生项目" | 工业极简 + 精密感，匹配风洞模型品质 |
| 缺少拍照/展示模式 | "没法炫耀" | 内置"展示模式"——一键最佳视觉效果 |
| 字体太小/太密 | "看不清，也不好看" | 舒适的阅读体验，排版有呼吸感 |

### Logo 上传体验
| 问题 | 用户感受 | zcritical 要求 |
|------|---------|---------------|
| 上传过程不透明 | "传了没？传到哪了？" | 整个过程像"定制装备"一样有完整的仪式感 |
| 失败无提示 | "为什么还是旧的？" | 不失败的体验——失败也要有面子 |
| 无预览/无创意引导 | "传什么好呢？" | 提供创意灵感——预览效果、内置模板 |

### 错误/异常体验
| 问题 | 用户感受 | zcritical 要求 |
|------|---------|---------------|
| 静默失败 | "点了一下没反应" | 所有操作反馈像精密设备一样可靠 |
| 错误信息是技术术语 | "CRC32 校验失败" | 翻译成用户语言 + 优雅的恢复引导 |
| 没有恢复路径 | "连不上，然后呢？" | 永远给用户下一步的选择 |

## 二、核心使用场景分析（基于正确产品定位）

### 场景 1: 桌面展示（日常状态）

```
用户: 风洞模型摆在桌上，LED常亮，风扇低速旋转，车模在里面
App: 后台运行或偶尔打开看看
操作: 几乎不操作——一切默认状态就好

设计要求:
- App 打开后自动连接，进入"仪表盘"视图
- 首页默认为"状态显示屏"而非"操作面板"
- 核心信息一目了然：当前模式、风速、颜色预览
- 设计足够精致，值得截图分享
```

### 场景 2: 把玩/调试（深度互动）

```
用户: 拿了一杯咖啡坐到桌前，拿出车模放进风洞
  → 打开 App → 看到了设备状态
  → 切换到"极速模式"看看车模在高速气流下的效果
  → 换个酷炫的红色 LED
  → 风速调低一点，再放一辆车进去对比
操作: 双手拿手机，有 5-10 分钟时间慢慢调

设计要求:
- 场景模式一键切换（预设风速+灯光+音效联动）
- 切换到自定义模式后，可以精细调整每个参数
- 调整时的视觉反馈要有"精密仪器"的感觉
- 颜色选择器应该是完整的色轮（不是 14 个预设方块）
```

### 场景 3: 上传个性化 Logo（仪式感操作）

```
用户: 做了一个车队 Logo，想传到风洞屏幕上
  → 打开 App → 进入 "定制" 页面
  → 选择图片 → 预览（240×240 的圆形/方形裁剪）
  → 看到自己设计的图案出现在预览中 "嗯，就是这个感觉"
  → 点击上传 → 进度动画 → "正在写入设备..."
  → 完成 → 设备屏幕亮出新 Logo → App 显示 "你的风洞，你的印记"
操作: 全神贯注，3-5 分钟，期待结果

设计要求:
- 完整的线性流程：选择 → 预览 → 裁剪 → 确认 → 上传 → 完成
- 每一步都是体验的一部分，不赶时间
- 上传完成后有"亮相"的仪式感
- 提供内置模板/字体，让不会设计的人也能做出好看的 Logo
```

### 场景 4: 朋友面前展示（社交时刻）

```
用户: 朋友来家里，看到桌上的风洞模型
  "哎这个酷，能用手机控制？"
  → 用户打开 App → 设备自动连上
  → 切换到 "展示模式" → 全屏显示当前状态
  → 演示调风速、换颜色 → LED 在深色桌面上效果极佳
  "还能上传自己的 Logo！"
  → 展示之前上传的车队 Logo

设计要求:
- 展示模式下 UI 极简——信息呈现像艺术品一样
- 切换操作干净利落——不要有加载/等待打扰展示
- 支持横屏展示（手机架在桌边）
```

### 场景 5: 初次开箱（第一次使用）

```
用户: 刚拿到风洞模型和配套 App
  → 打开 App → 欢迎引导（2-3 页）
  → 提示开启蓝牙 → 自动搜索设备
  → "发现你的风洞模型" → 点击连接 → 连接成功动画
  → 进入首页 → 风洞模型开始低速旋转以示响应
  → 小提示引导用户尝试调整风速
操作: 好奇、探索、期待"这玩意能干什么"

设计要求:
- 首次使用 3 步引导，不超过 3 页
- 连接设备的过程本身就是开箱体验的延伸
- 连接成功后设备有可见响应（LED 闪一下/风扇转一下）
- 首次使用的 1 分钟内让用户"感受到"这个产品的魅力
```

## 三、交互设计原则（匹配桌面把玩场景）

### 原则 1: 操作有"手感"

```
旧(RideWind): 拖滑块 → 延迟 → 硬件响应 → UI 更新（工业遥控器感）
新(zcritical): 拖滑块 → UI 即刻响应 + 细腻动画 → 硬件确认 → 震动反馈
```

每一次操作都像一个精密旋钮——有刻度感、有回馈、有声音（可选）。

### 原则 2: 场景化 > 参数化

不要给用户 8 个独立参数。给 3-5 个场景模式 + 自定义模式。

| 预设模式 | 风速 | LED | 音效 | 意图 |
|---------|------|-----|------|------|
| 极速模式 | 100% | 红色脉冲 | 高亢引擎声 | "感受风的威力" |
| 巡航模式 | 40% | 蓝色渐变 | 低沉巡航声 | "优雅展示" |
| 展览模式 | 20% | 全彩流转 | 关 | "安静地好看" |
| 自定义 | 任意 | 任意 | 任意 | "我的专属配置" |

### 原则 3: 状态可视化——像仪表盘一样

- 连接状态: 顶部状态栏，像跑车内饰的指示灯
- 风速: 大号数字 + 风洞风格的仪表盘图形
- LED 颜色: 实时预览色块（和硬件同步的颜色）
- 模式: 当前选中模式高亮，一个醒目的名称

### 原则 4: Logo 上传——一场小仪式

上传 Logo 不是功能操作，是"定制仪式"：
- 引导语: "为你的风洞选择专属印记"
- 预览: 240×240 的精确预览在 App 中显示
- 创意工具: 内置几种字体模板（"不需要你会PS"）
- 上传完成后: 设备屏幕亮起新 Logo + App 显示完成动画
- 可以拍照分享

### 原则 5: 展示模式——"让产品自己说话"

App 有一个"展示模式":
- UI 极简化（只留关键信息）
- 全屏显示当前设备状态 + 视觉效果
- 适合横屏放置（手机放在支架上）
- 可以配合设备做出灯光秀效果

### 原则 6: 错误提示有温度和个性

| 技术事件 | ❌ 技术提示 | ✅ 用户提示 |
|---------|-----------|-----------|
| BLE 超时 | "Connection timeout (30s)" | "看来风洞有点远，再试一次？靠近一点应该就OK了" |
| CRC 校验失败 | "CRC32 mismatch" | "传输中出了点小问题，我们重新传一次" |
| Logo 槽位满 | "All logo slots occupied" | "需要替换一个已有 Logo 来放新的，选择要替换的吧" |
| BLE 权限未开启 | "Bluetooth permission denied" | "需要开启蓝牙才能连接风洞模型～去设置打开就行"

## 四、视觉设计方向

### 设计语言: "工业极简 × 精密感"

灵感来源: 航空仪表盘、高精度仪器控制面板、F1 赛车方向盘 UI

### 配色方向

| 用途 | 色系 | 感觉 |
|------|------|------|
| 背景(深色) | 深灰黑 `#1A1A2E` `#121212` | 稳重、专业、适合展示 |
| 背景(浅色) | 纯白/浅灰 | 白天使用 |
| 主色调 | 科技蓝 `#4A90D9` 或 赛车红 `#E74C3C` | 品牌色，待你决定 |
| 风速/动力 | 蓝→青→红渐变（低→高） | 速度的能量感 |
| LED 预览 | 实时同步硬件颜色 | 所见即所得 |
| 成功/连接 | 绿色脉冲 `#2ECC71` | 确认感 |
| 错误/断开 | 琥珀/红 `#E74C3C` | 警告但不恐慌 |

### 首页布局方向

```
┌─────────────────────────────┐
│  🔵 ZCritical  已连接        │  ← 状态栏
├─────────────────────────────┤
│                             │
│     ┌─────────────────┐     │
│     │                 │     │  ← 设备状态仪表盘区域
│     │   风速: 45%     │     │     (大字号，核心视觉)
│     │   ████████░░    │     │
│     │   模式: 巡航    │     │
│     │   ● 极速 ○ 展览 │     │  ← 模式快捷切换
│     │                 │     │
│     └─────────────────┘     │
│                             │
│   [LED颜色] [音量] [Logo]   │  ← 功能入口区
│                             │
├─────────────────────────────┤
│  🏠 首页  🎨 定制  🎮 场景  ⚙️  │  ← 底部导航
└─────────────────────────────┘
```

### 动效原则

- 速度变化 → 数字平滑过渡动画（像车载仪表盘）
- 模式切换 → 流畅的颜色渐变 + 风扇转速变化可视化
- 连接成功 → 脉冲动画，像"设备苏醒"
- 操作反馈 → 微妙的弹性动画（按下 → 回弹）
- Logo 更新 → 屏幕擦除 + 新图案浮现的过渡动画
- **动效是体验的一部分，让产品"活"起来**

## 五、UX 决策 vs 架构决策 冲突处理

当用户体验和架构纯粹性冲突时：

| 冲突场景 | 处理 |
|---------|------|
| 用户体验需要跳过一层抽象 | **允许跳过**，加注释说明为什么 |
| 用户体验需要"乐观更新"（UI先行） | **允许**，在 Provider 层用 optimistic state |
| 用户体验需要合并两个 Provider 的状态显示 | **允许**，建一个组合 Provider |
| 用户体验需要减少操作步骤 | **优先减少步骤**，架构可后续优化 |

**一句话**: 架构为体验服务，不是反过来。

## 六、UX 检查清单（每个功能上线前必过）

- [ ] 用户在 3 秒内能找到主要功能吗？
- [ ] 每个操作都有即时视觉反馈吗？（< 200ms）
- [ ] 操作区域 ≥ 48×48dp 吗？
- [ ] 任何等待超过 500ms 的操作有进度指示吗？
- [ ] 任何可能的失败都有用户友好的提示吗？
- [ ] 每个错误提示都告诉用户"下一步做什么"了吗？
- [ ] 深色模式下可读吗？
- [ ] 在最大和最小主流屏幕上都显示正常吗？
- [ ] 新用户不用看说明书就会用吗？
- [ ] 所有文案是用户语言而非技术术语吗？
